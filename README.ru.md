**Саб-агенты (subagents) — это «запустил и забыл». Taskflow (рабочие процессы) — запускают, разветвляют, приостанавливают, шлюзуют, возобновляют и сохраняют себя как команду.**

Вам уже знаком встроенный инструмент саб-агентов — `task` / `tasks` / `chain`. `pi-taskflow` использует ту же сокращённую запись (shorthand), поэтому ваши текущие делегирования мгновенно становятся **отслеживаемыми, возобновляемыми и сохраняемыми как однословная команда `/tf:<name>`**. Когда сокращённой записи становится недостаточно, полный DSL (предметно-ориентированный язык) предоставляет настоящий DAG (ориентированный ациклический граф): динамическое разветвление (fan-out) на десятки элементов, условную маршрутизацию, шлюзы качества (quality gates), утверждения человеком (human approvals), повторные попытки (retries) и жёсткий лимит затрат (spend ceiling).

И всё это время **до вашего разговора доходит только финальная фаза.** Все промежуточные транскрипты остаются в рантайме (runtime), никогда не попадая в окно вашего контекста (context window).

## Зачем это нужно

Вот стена, о которую вы разбиваетесь, используя «сырые» саб-агенты: вы описываете многошаговый план в прозе, модель выводит его заново при каждом запуске, промежуточные транскрипты затапливают ваш контекст, а при первом сбое вызова модели начинаете с нуля. Ни переиспользования, ни восстановления, ни структуры.

`pi-taskflow` **переносит план из промпта (prompt) в декларативное определение (declarative definition).** Рантайму принадлежит DAG, циклы, повторные попытки и промежуточное состояние. Вы объявляете конвейер (pipeline) один раз и запускаете его сотню раз — по имени.

> Когда задача требует двенадцати шагов с ветвящимся разветвлением и проверочным шлюзом, вам нужна оркестрация (orchestration) — а не удачный промпт.

| | саб-агент (встроенный) | **pi-taskflow** |
|---|---|---|
| **Кто управляет** | модель, шаг за шагом | рантайм, на основе определения |
| **Топология** | цепочка / плоский параллелизм | **DAG со слоистой конкурентностью + маршрутизация** |
| **Промежуточные результаты** | в окне вашего контекста | **в рантайме — не в вашем контексте** |
| **Масштаб** | горстка задач | **динамический `map` (разветвление) на десятки элементов** |
| **Переиспользование** | переописывается каждый раз | **сохраняется как `/tf:<name>`** |
| **Возобновление** | ✗ | **✓ между сессиями — кешированные фазы пропускаются автоматически** |
| **Шлюзы качества** | ✗ | **фазы `gate`, останавливающие выполнение при `VERDICT: BLOCK`** |
| **Условная маршрутизация** | ✗ | **охранители `when` + OR-объединение `join: any`** |
| **Отказоустойчивость** | ✗ | **`retry` на фазу + авто-повтор при временных ошибках** |
| **Человек в цикле** | ✗ | **фазы `approval` (утвердить / отклонить / изменить)** |
| **Контроль затрат** | ✗ | **`budget` на весь запуск (лимиты USD / токенов)** |
| **Композиция** | ✗ | **фазы `flow` запускают сохранённые под-процессы (sub-flows)** |
| **Прогресс в реальном времени** | непрозрачно во время выполнения | **отрисовка DAG в реальном времени с таймингом и стоимостью** |
| **Эргономика** | встроенный JSON каждый раз | **сокращённая запись (`task`/`tasks`/`chain`) *или* DSL** |

Это не замена инструменту саб-агентов. Это наделяет ваши саб-агенты DAG, памятью и именем.

**Как выбирать:**

- **`@pi-agents/orchid`** — самый функциональный оркестратор в экосистеме (DAG + worktrees + цикл Ralph + почтовый ящик агента) — но его DSL — это *фиксированный* 9-фазный пайплайн, он несёт зависимости времени выполнения + jiti и находится в бета-версии. Выбирайте `pi-taskflow`, когда хотите **определить свой собственный граф** (а не принять чужой) с **нулевыми зависимостями** и установкой одной командой.
- **`pi-crew` / `ultimate-pi`** идут дальше — изоляция worktree, долговременные асинхронные команды, многоуровневое управление. Если вам нужно лёгкое, декларативное решение с нулевыми зависимостями — это этот проект.
- **`@zhushanwen/pi-workflow`** ближе всего по духу и тоже без зависимостей, но вы создаёте workflow как **JavaScript-скрипты**. **Декларативный JSON DSL** от `pi-taskflow` безопаснее и удобнее для аудита, а его **возобновление по хешу ввода на уровне фаз** более гранулярно, чем дедупликация кеша вызовов.
- **`@fiale-plus/pi-rogue-orchestration`** имеет настоящий **цикл-до-завершения** (функция, которой у `pi-taskflow` пока нет). Если ваша задача — «продолжать, пока цель не достигнута», стоит присмотреться; `pi-taskflow` предназначен для *структурированных, ветвящихся* пайплайнов.
- **`pi-subagents` / `@gotgenes/pi-subagents`** — зрелый выбор для ad-hoc делегирования «запустить рецензента на этот diff» и фоновых задач. `pi-taskflow` — для случаев, когда эти делегирования должны стать *повторяемым, возобновляемым пайплайном*.
- **`pi-pipeline` / `pi-agent-flow`** поставляются с *фиксированными, мнениями* потоками. `pi-taskflow` — это *чистый холст*: вы (или модель) объявляете граф, который подходит под задачу.

> Честная однострочная характеристика: **`pi-taskflow` — единственное расширение Pi, предоставляющее декларативный, возобновляемый DAG-пайплайн для саб-агентов, сохраняемый как однословная команда, с нулевыми зависимостями времени выполнения и изоляцией контекста (context isolation) по умолчанию.** Известные пробелы, которые закрываются в следующих версиях: цикл-до-завершения, изоляция worktree и неблокирующие фоновые запуски (см. [`STRATEGY.md`](./STRATEGY.md)).

## Старт за 30 секунд

**1. Установка** — одна команда:

```bash
pi install npm:pi-taskflow
```

> **Опционально:** выполните `/tf init` один раз, чтобы сопоставить роли моделей (model roles) 18 встроенных агентов
> (`fast`, `strong`, `thinker`, …) с вашими моделями — интерактивный выбор.
> Пропустите — агенты будут использовать модель Pi по умолчанию. См. [Роли моделей](#model-roles).

**2. Запуск** — просто попросите модель в сессии Pi:

> *Запусти цепочку: сначала изучи auth flow, затем суммируй результаты.*

Модель вызовет инструмент `taskflow` автоматически. Вы получите прогресс в реальном времени, тайминг каждого шага, стоимость в токенах и сохранённую запись запуска — **те же усилия, что и встроенный инструмент, но теперь отслеживаемые и возобновляемые.**

**3. Сохранение** — скажите *«сохрани это»* — и у вас появится `/tf:<name>` навсегда.

Вот и всё. Вы можете запустить свой первый workflow до того, как остынет кофе — без написания единого определения фазы.

### Сокращённая запись (shorthand) — та же форма, что у встроенного инструмента

```jsonc
// Одиночный — один агент, одна задача
{ "task": "Summarize the architecture of src/", "agent": "explorer" }

// Параллельный — запустить несколько сразу, результаты сливаются
{ "tasks": [
  { "task": "Audit auth in src/api",             "agent": "analyst" },
  { "task": "Audit input validation in src/api", "agent": "analyst" }
] }

// Цепочка — последовательно; каждый шаг видит вывод предыдущего
{ "chain": [
  { "task": "List the public API of src/lib", "agent": "scout" },
  { "task": "Write docs for:\n{previous.output}", "agent": "writer" }
] }
```

Поле `agent` опционально (по умолчанию — первый найденный агент). Добавьте `name`, чтобы пометить запуск и получить возможность сохранить его как команду.

## Смотрите, как это работает

Это не мокап. **Это stdout реального запуска** — flow `self-improve`, который пишет и проверяет свои собственные тестовые наборы, пойманный на лету шлюзом качества:

```
⊗ taskflow self-improve  6/7 · blocked · $0.095
    ✓ discover            agent   deepseek-v4-flash  10t ↑38k ↓6.7k $0.011
  ┌ ✓ write-runner-tests  agent   claude-sonnet-4-6  10t ↑13 ↓6.6k $0.020
  ├ ✓ write-store-tests   agent   claude-sonnet-4-6  10t ↑11 ↓10k $0.018
  ├ ✓ write-agents-tests  agent   claude-sonnet-4-6  10t ↑28 ↓13k $0.030
  └ ✓ fix-stability       agent   claude-sonnet-4-6  10t ↑13 ↓3.9k $0.012
    ✓ verify              gate    BLOCK 3 type errors in test files  deepseek-v4-flash
    ⊘ report              reduce  skipped · Gate blocked  ↳ fix-stability
```

**Макет *и есть* DAG.** Никаких дашбордов, никаких логов для grep — вы читаете индикатор прогресса и понимаете весь пайплайн:

- **Заголовок** — `⊗` = заблокирован (шлюз остановил); обработано `6/7` фаз; общая стоимость `$0,095`.
- **Иконки статуса** — `✓` готово · `◐` выполняется · `✗` ошибка · `⊘` пропущено · `○` ожидает.
- **Рельсы `┌ ├ └`** — фазы в одном слое DAG, выполняющиеся конкурентно. Четыре задачи `write-*`/`fix-stability` разветвляются из `discover`. Пустой желоб = однофазовый слой.
- **`↳`** — длинная, перепрыгивающая слой зависимость. `report` зависит от соседнего `verify` *и* от `fix-stability` на два слоя назад, поэтому аннотировано только это ребро-прыжок.
- **Шлюз** — `verify` выдал `VERDICT: BLOCK`, поэтому рантайм пропустил `report` и завершил запуск как `blocked`, выведя причину прямо в строке.
- **Детали** — на фазу: модель, количество токенов (`↑`вход `↓`выход), стоимость, тайминг. Фазы с разветвлением также показывают прогресс подзадач (`3/15 2✗ 8▸`).

## Переходите к декларативному подходу

Сокращённая запись — это ваш входной трамплин. DSL — это то, где `pi-taskflow` раскрывает свой потенциал: динамическое разветвление, структурированная маршрутизация и шлюзы качества.

### Разветвление и свёртка (fan out and reduce)

```jsonc
{
  "name": "summarize-files",
  "description": "Discover files, summarize each, produce one report",
  "args": { "dir": { "default": "." } },
  "concurrency": 8,
  "phases": [
    { "id": "discover", "type": "agent", "agent": "scout",
      "task": "List source files under {args.dir} (non-recursive).\nOutput ONLY a JSON array [{\"file\":\"\"}]. No prose.",
      "output": "json" },
    { "id": "summarize", "type": "map",
      "over": "{steps.discover.json}", "as": "item", "agent": "scout",
      "task": "Read {item.file} and give a one-sentence summary.",
      "dependsOn": ["discover"] },
    { "id": "report", "type": "reduce", "from": ["summarize"], "agent": "writer",
      "task": "Combine into a short overview:\n{steps.summarize.output}",
      "dependsOn": ["summarize"], "final": true }
  ]
}
```

1. **`discover`** перечисляет каждый файл и выводит JSON-массив.
2. **`summarize`** — это `map`: он разветвляется на одного саб-агента на файл, с ограничением до 8 конкурентных, при этом `{item.file}` привязан к каждому пути.
3. **`report`** — это `reduce`: он объединяет все саммари в один чистый обзор.

Промежуточные саммари никогда не попадают в ваш контекст. Рантайм владеет ими; вы получаете отчёт. **Сохраните один раз → `/tf:summarize-files dir=src` навсегда.**

### Маршрутизация, шлюз, повторы, утверждение и лимит затрат

```jsonc
{
  "name": "triage-and-fix",
  "budget": { "maxUSD": 1.5 },
  "phases": [
    { "id": "triage", "type": "agent", "agent": "analyst", "output": "json",
      "task": "Classify the bug. Output ONLY {\"severity\":\"high\"} or {\"severity\":\"low\"}." },
    { "id": "deep",  "when": "{steps.triage.json.severity} == high", "dependsOn": ["triage"],
      "agent": "executor-code", "task": "Root-cause and patch it.",
      "retry": { "max": 2, "backoffMs": 500 } },
    { "id": "quick", "when": "{steps.triage.json.severity} == low",  "dependsOn": ["triage"],
      "agent": "executor-fast", "task": "Apply the quick fix." },
    { "id": "approve", "type": "approval", "join": "any", "dependsOn": ["deep", "quick"],
      "task": "Review the fix before it ships." },
    { "id": "ship", "type": "agent", "dependsOn": ["approve"],
      "task": "Open a PR with the change.", "final": true }
  ]
}
```

- **`when`** направляет в `deep` *или* `quick` на основе JSON-результата триажа — другая ветка пропускается.
- **`join: "any"`** позволяет `approve` сработать сразу по завершении любой из запущенных веток (OR-объединение).
- **`retry`** перезапускает ненадёжный патч с задержкой (backoff); **`budget`** останавливает весь запуск, если он становится слишком дорогим.
- **`approval`** приостанавливается для человека (утвердить / отклонить / изменить) перед финальным `ship`.

Никакого скриптования. Никакого `eval`. Просто данные, которые выполняет рантайм — достаточно безопасно, чтобы выполнять определения, сгенерированные LLM, напрямую.

## Типы фаз

| type | что делает | обязательные поля |
|------|-----------|-------------------|
| `agent` | один саб-агент выполняет одну задачу | `task` |
| `parallel` | запускает `branches[]` конкурентно | `branches` (массив `{task, agent?}`) |
| `map` | **разветвление** по массиву — один саб-агент на элемент, привязан `{item}` | `over`, `task` |
| `gate` | шлюз качества/рецензии, может **остановить поток** | `task` |
| `reduce` | агрегация выводов фаз из `from[]` в один | `from`, `task` |
| `approval` | **пауза «человек в цикле»** — утвердить / отклонить / изменить | — |
| `flow` | запускает **сохранённый под-процесс** как одну фазу (композиция) | `use` |
| `loop` | **итерация задачи до завершения** — повторяет тело, пока не выполнено условие, не достигнута сходимость или не превышен лимит | `task`, `until` |
| `tournament` | **N вариантов соревнуются**, судья выбирает лучший (или агрегирует) | `task` \| `branches` |

### Общие поля фаз

Каждая фаза требует уникального `id` и `type` (по умолчанию `agent`). В дополнение к полям, специфичным для типа:

| Поле | Значение |
|---|---|
| `agent` | Агент для запуска (по умолчанию — первый найденный агент) |
| `dependsOn` | ID фаз, которые эта фаза ожидает — строит DAG |
| `join` | `"all"` (по умолчанию) ждёт все зависимости; `"any"` — OR-объединение |
| `when` | Условный охранитель (guard) — пропустить, если выражение ложно |
| `retry` | `{ max, backoffMs?, factor? }` — повтор при сбое саб-агента |
| `output` | `"text"` (по умолчанию) или `"json"` (открывает доступ к `{steps.ID.json}`) |
| `model` / `thinking` / `tools` | Переопределения на уровне фазы для саб-агента |
| `cwd` | Рабочая директория для саб-агента |
| `concurrency` | Лимит разветвления для `map` / `parallel` (переопределяет умолчание flow) |
| `final` | Помечает фазу, несущую результат (иначе побеждает последняя фаза) |
| `optional` | Сбой здесь **не прерывает** запуск |
| `use` / `with` | (`flow`) имя сохранённого под-процесса + его аргументы |
| `cache` | `{ scope, ttl?, fingerprint? }` — межзапускное мемоизация (см. ниже) |

Ключи уровня flow: `name`, `description`, `args`, `concurrency` (по умолчанию 8), `agentScope` и `budget: { maxUSD?, maxTokens? }`.

### Управление потоком и надёжность

- **`when`** — пропустить фазу, если выражение ложно (falsy). Поддерживает `{refs}`, `== != < > <= >=`, `&& || !`, скобки и строки/числа в кавычках. В паре с `join: "any"` на фазе слияния даёт полноценную маршрутизацию if/else. Ошибки парсинга **пропускают фазу** (fail open).
- **`join: "any"`** — OR-объединение: фаза запускается, как только *одна* зависимость завершится (по умолчанию `"all"` ждёт все).
- **`retry`** — `{ "max": 2, "backoffMs": 500, "factor": 2 }` повторяет сбойного саб-агента с фиксированной или экспоненциальной задержкой; использование суммируется, количество попыток показывается как `↻N` в TUI. Временные ошибки провайдера (rate-limit / 5xx / timeout) **автоматически повторяются даже без явной политики**; жёсткие ошибки — нет.
- **`approval`** — пауза для человека (Утвердить / Отклонить / Изменить). Отклонение останавливает поток; Изменение вставляет введённую заметку как вывод фазы для последующих шагов. В неинтерактивных запусках автоматически утверждается.
- **`flow`** — `{ "type": "flow", "use": "deep-research", "with": { "topic": "{item}" } }` запускает сохранённый flow как фазу (рекурсия обнаруживается и отклоняется).

### Цикл-до-завершения (`loop`)

Некоторая работа по своей природе итеративна — дорабатывать черновик, пока рецензент не удовлетворён, повторять и улучшать, пока тесты не пройдут, сходиться к ответу. Фаза `loop` перезапускает одно тело задачи до тех пор, пока не выполнится условие остановки:

```jsonc
{
  "id": "refine",
  "type": "loop",
  "task": "Improve this draft (iteration {loop.iteration}). Previous attempt:\n{loop.lastOutput}\n\nReturn JSON {\"draft\":\"…\",\"done\":true|false}.",
  "until": "{steps.refine.json.done} == true",   // вывод самой итерации доступен здесь
  "output": "json",
  "maxIterations": 6,        // по умолчанию 10, жёсткий лимит 100 — цикл ВСЕГДА завершается
  "convergence": true        // по умолч.: остановиться досрочно, если вывод итерации идентичен предыдущему
}
```

- **Локальные переменные тела** — задача может читать `{loop.iteration}` (начиная с 1), `{loop.lastOutput}` (вывод предыдущей итерации) и `{loop.maxIterations}`, чтобы строить на основе своей предыдущей работы; все три также доступны в условии `until`.
- **`until`** — вычисляется после каждой итерации, при этом вывод итерации доступен как `{steps.<thisId>.output}` / `.json`. Те же операторы, что и у `when`. Цикл останавливается, как только условие истинно (truthy).
- **Всегда завершается.** Четыре независимых механизма остановки: `until` истинно, **сходимость** (convergence — фиксированная точка: вывод идентичен предыдущей итерации), **`maxIterations`** (жёсткий лимит 100), или **сбойная итерация** (фаза завершается ошибкой с сохранённым частичным выводом). Некорректный `until` **останавливает** цикл, а не зацикливает его бесконечно (fail-safe) и выводит предупреждение на фазе.
- TUI показывает `↻N` с причиной остановки (`done` / `converged` / `max` / `failed`); использование суммируется по итерациям. Как и `gate`/`approval`, `loop` **исключён из `cross-run` кеша** (каждый запуск должен выполнять итерации заново).

### Турнир (`tournament`)

Для открытых задач лучший результат часто достигается генерацией нескольких кандидатов и выбором сильнейшего — best-of-N с судьёй, в одной декларативной фазе:

```jsonc
{
  "id": "headline",
  "type": "tournament",
  "task": "Write a punchy headline for this launch post.",
  "variants": 4,                    // породить 4 конкурента одной ЗАДАЧИ (по умолч. 3, макс. 20)
  "judge": "Pick the headline with the strongest hook and clearest promise.",
  "judgeAgent": "reviewer",          // опционально; по умолчанию — агент фазы
  "mode": "best"                     // "best" (по умолч.) | "aggregate"
}
```

- **Конкуренты** — либо `variants: N` копий одной `task` (разнообразие за счёт недетерминизма модели), либо отдельные `branches: [{task, agent?}, …]`, когда нужно столкнуть *разные подходы* друг с другом.
- **Судья (judge)** — после разветвления один агент-судья видит каждый вариант (пронумерованный) плюс вашу рубрику `judge` и выбирает победителя через строку `WINNER: <n>` или `{"winner": n}`. Нераспознаваемый вердикт **переключается на вариант 1** (fail open); сбой судьи тоже откатывается — работа никогда не теряется.
- **`mode`** — `best` возвращает победивший вариант **дословно**; `aggregate` возвращает **синтезированный** ответ судьи, объединяющий сильнейшие части.
- **Короткое замыкание:** если выжил только один конкурент, он побеждает без вызова судьи; если все провалились — фаза завершается ошибкой. TUI показывает `⚑ N→#k`; использование суммирует варианты + судью. Как и `gate`, **исключён из `cross-run` кеша**.
- **`budget`** — общепоточный потолок `{maxUSD, maxTokens}`; при превышении ожидающие фазы пропускаются, а разветвление прекращает порождать новые, завершая запуск как `blocked`.
- **Сторожевой таймер бездействия (idle watchdog)** — саб-агент, замолчавший на 5 минут, считается зависшим и убивается (SIGTERM → SIGKILL), так что один зависший дочерний процесс не может заморозить весь поток.

### Межзапускная мемоизация (`cache`)

Каждая фаза уже адресуется по содержимому (content-addressed): в пределах одного **возобновления** запуска фаза, чьи разрешённые входные данные не изменились, пропускается. `cache` расширяет это переиспользование на **независимые запуски** — если какой-либо предыдущий запуск вычислил фазу с идентичным хешем ввода, её результат переиспользуется за **$0,00**.

```jsonc
{
  "id": "analyze-auth",
  "task": "Summarize how the auth module works.",
  "context": ["src/auth/**/*.ts"],
  "cache": {
    "scope": "cross-run",                 // "run-only" (по умолч.) | "cross-run" | "off"
    "ttl": "6h",                          // необ. макс. возраст; при превышении попадание считается промахом
    "fingerprint": ["git:HEAD", "glob:src/auth/**/*.ts"]  // свернуть состояние мира в ключ
  }
}
```

- **`scope`** — `"run-only"` (по умолчанию) — это историческое поведение (возобновление только в пределах запуска). `"cross-run"` включает фазу в постоянное хранилище. `"off"` отключает переиспользование полностью (даже в пределах запуска) для отладки.
- **Свежесть — главное.** Ключ кеша уже включает промпт, элементы `over` и любые файлы `context` (прочитанные в задачу). `fingerprint` добавляет *неявные* входные данные в ключ, так что «мир изменился» становится промахом кеша: `git:HEAD`, `glob:<pat>` (размер+mtime), `glob!:<pat>` (хеш содержимого), `file:<path>`, `env:<NAME>`. `ttl` (`30m`/`6h`/`7d`) — временная подстраховка.
- **Честное ограничение:** саб-агент, который читает файл, не объявленный в `context`/`fingerprint`, всё ещё может получить устаревшее попадание `cross-run`. Поэтому по умолчанию стоит `run-only` и фазы `gate`/`approval` **запрещены** для `cross-run` (они должны давать свежий результат в каждом запуске). Включайте только для фаз, чей вывод является функцией объявленных входных данных.
- Кеш живёт в `.pi/taskflows/cache/` (в gitignore). Очистите через `action: "cache-clear"`. Полное обоснование: [`docs/rfc-cross-run-memoization.md`](./docs/rfc-cross-run-memoization.md).

### Фазы-шлюзы (gate) — контроль качества

`gate` запускает агента для рецензирования вывода предыдущих фаз и может **блокировать остальную часть workflow.** Завершите задачу шлюза запросом вердикта, который сможет прочитать рантайм:

- финальная строка `VERDICT: PASS` или `VERDICT: BLOCK` (также принимает `OK`, `FAIL`, `STOP`, `REJECT`, `HALT` — последнее вхождение побеждает), или
- JSON вида `{"continue": false, "reason": "missing auth checks"}` / `{"verdict": "block", "reason": "..."}`.

При **BLOCK** последующие фазы пропускаются, и запуск завершается как `blocked` с указанием причины. **Неоднозначный вывод открывает проход (fail open — считается PASS)** — шлюз никогда не остановит ваш поток случайно.

```
Review the audit below. If any endpoint is missing auth, end with
"VERDICT: BLOCK" and a one-line reason; otherwise end with "VERDICT: PASS".

{steps.audit.output}
```

## Интерполяция и выражения

| плейсхолдер (placeholder) | разрешается в |
|---|---|
| `{args.X}` | аргумент вызова |
| `{steps.ID.output}` | текстовый вывод предыдущей фазы |
| `{steps.ID.json}` | вывод предыдущей фазы, разобранный как JSON (или `{steps.ID.json.field}`) |
| `{item}` / `{item.field}` | текущий элемент внутри фазы `map` |
| `{previous.output}` | вывод непосредственно вышестоящей фазы |

Грамматика условий (для `when`): `== != < > <= >=`, `&& || !`, скобки, строки/числа в кавычках и любые ссылки `{...}` — например, `"when": "{steps.triage.json.route} == deep && {args.force} != true"`.

> Ссылка на `{steps.X}`, не объявленный в `dependsOn`, — это **жёсткая ошибка валидации** — рантайм ловит самую распространённую багу пайплайна до запуска единого агента.

## Команды

Сохранённые flow становятся CLI-шорткатами. Все команды выполняются в сессии Pi:

| Команда | Что делает |
|---|---|
| `/tf list` | Список всех сохранённых flow |
| `/tf run <name> [args]` | Запустить сохранённый flow (например, `/tf run summarize-files dir=src`) |
| `/tf show <name>` | Показать определение flow |
| `/tf runs` | Просмотр недавней истории запусков (интерактивный TUI) |
| `/tf resume <runId>` | Продолжить приостановленный/прерванный запуск — кешированные фазы пропускаются автоматически |
| `/tf init` | **Интерактивно сопоставить роли моделей** с вашими включёнными моделями (записывает `~/.pi/agent/settings.json`) |
| `/tf:<name> [args]` | Шорткат — запускает flow одним нажатием |

Действия инструмента (используются моделью): `run` (встроенное `define` или сохранённое `name`), `save`, `resume`, `list`, `init`.

## Возобновление между сессиями

Запуск taskflow не привязан к вашей сессии. Каждая завершённая фаза записывается на диск, поэтому запуск, который прервался (или который вы остановили), можно продолжить позже командой `/tf resume <runId>` — **кешированные фазы пропускаются автоматически**, и только оставшаяся работа расходует токены.

<div align="center">
<img src="./assets/resume.png" alt="A run fails midway in session 1; in session 2 /tf resume skips the cached phases and only re-runs the failed phase and what follows" width="900">
</div>

Возобновление привязано к хешу ввода каждой фазы — если вывод вышестоящей фазы изменился, зависимые фазы перезапускаются; если ничего не изменилось — они переиспользуются. Ни одно другое расширение Pi не делает этого между сессиями.

## Хранилище

```
.pi/taskflows/<name>.json          # определения в рамках проекта (commit для общего доступа)
~/.pi/agent/taskflows/<name>.json  # определения в рамках пользователя
.pi/taskflows/runs/<runId>.json    # состояние запуска для возобновления (добавьте в gitignore)
```

> Закоммитьте `.pi/taskflows/` — и вся ваша команда делится пайплайнами: никакой синхронизации конфигов, никакого онбординга. Состояние запуска записывается атомарно и защищается файловой блокировкой без внешних зависимостей, так что конкурентные запуски никогда не повредят индекс.

Область обнаружения агентов (через `agentScope` в определении flow):

| значение | откуда обнаруживает агентов |
|---|---|
| `"user"` (по умолч.) | `~/.pi/agent/agents/*.md` |
| `"project"` | `.pi/agents/*.md` (поднимается по дереву) |
| `"both"` | пользователь + проект; проект побеждает при коллизии имён |

## Агенты

Taskflow поставляется с **18 встроенными агентами** — каждый представляет собой `.md`-файл с настроенным системным промптом, уровнем мышления (thinking) и набором инструментов. Вы можете ссылаться на них по `name` в любой фазе или сокращённой записи сразу после установки. Никакой настройки не требуется.

### Состав встроенных агентов

| Агент | Роль | Thinking | Роль модели по умолч. |
|---|---|---|---:|---|
| `executor` | Внедрение запланированных изменений кода | high | `{{fast}}` |
| `executor-fast` | Тривиальные исправления (≤2 файла, ≤50 строк) | off | `{{fast}}` |
| `executor-code` | Сложная многофайловая реализация | high | `{{strong}}` |
| `executor-ui` | Фронтенд / стилизация / визуальные изменения | high | `{{vision}}` |
| `scout` | Быстрая разведка кодовой базы и картографирование файлов | off | `{{fast}}` |
| `planner` | Создание плана реализации | high | `{{strong}}` |
| `analyst` | Анализ требований, обнаружение неоднозначностей | high | `{{thinker}}` |
| `critic` | Внутреннее сомнение в процессе рассуждения | xhigh | `{{thinker}}` |
| `reviewer` | Общая рецензия кода / архитектуры | high | `{{strong}}` |
| `risk-reviewer` | Риски бэкенда / инфраструктуры / БД / API | high | `{{reasoner}}` |
| `security-reviewer` | Уязвимости безопасности, аутентификация/криптография | xhigh | `{{reasoner}}` |
| `plan-arbiter` | Шлюз качества плана (сложные задачи) | high | `{{arbiter}}` |
| `final-arbiter` | Третейский судья при разногласиях рецензентов | xhigh | `{{arbiter}}` |
| `test-engineer` | Проектирование и реализация тестов | high | `{{fast}}` |
| `doc-writer` | Написание документации | off | `{{fast}}` |
| `recover` | Восстановление сессии после уплотнения (compaction) | low | `{{fast}}` |
| `verifier` | Запуск тестов, проверка результатов | off | `{{fast}}` |
| `visual-explorer` | Анализ метаданных дизайна Figma | high | `{{vision}}` |

Агенты выстроены слоями: **встроенный → пользовательский (`~/.pi/agent/agents/`) → проектный (`.pi/agents/`)**. Пользовательский или проектный агент с тем же `name` переопределяет встроенного — так вы можете настроить любого агента, не трогая пакет.

### Роли моделей (model roles)

Поле `model` каждого встроенного агента использует **ролевой плейсхолдер** (например, `{{fast}}`) вместо жёстко заданной строки провайдера. Это разделяет *намерение* и *реализацию* — вы сопоставляете роли с моделями один раз, и каждый агент адаптируется.

| Роль | Намерение | Типичная модель |
|---|---|---|
| `{{fast}}` | Дешёво и быстро — высокий объём, низкая ответственность | DeepSeek V4 Flash |
| `{{strong}}` | Сбалансированно — планирование, рецензия, умеренная сложность | MiMo v2.5 Pro |
| `{{thinker}}` | Глубокий анализ — требования, критика | DeepSeek V4 Pro |
| `{{arbiter}}` | Окончательное суждение — третейский суд, шлюзы качества планов | Qwen 3.7 Max |
| `{{vision}}` | Мультимодально — UI-работа, чтение дизайнов | MiniMax M3 |
| `{{reasoner}}` | Осторожное рассуждение — безопасность, риски | GLM 5.1 |

Без конфигурации агенты используют модель Pi по умолчанию. Чтобы сопоставить роли с реальными моделями, запустите интерактивную настройку:

```bash
/tf init
```

`/tf init` начинается с **меню действий**. Новички видят шорткат из 2 опций («Использовать рекомендованные настройки» / «Настроить каждую роль»). Вернувшиеся пользователи видят полное меню из 5 опций:

```
? What do you want to do with model roles?
  ❯ Use recommended defaults
    Configure each role
    Edit one role
    Show current roles
    Cancel
```

### Путь инструмента (`action="init"`)

Модель также может настраивать роли через инструмент `taskflow`:

| Режим | Поведение |
|---|---|
| `mode: "show"` (по умолч.) | Отчёт только для чтения о текущих `modelRoles`. Никогда не перезаписывает. |
| `mode: "apply-defaults"` + `force: true` | Записывает `RECOMMENDED_DEFAULTS` в `settings.json`, сохраняя устаревшие ключи. |
| `mode: "interactive"` | Запускает полное меню действий + поток выбора (требуется UI-сессия). |

> **Примечание о прекращении поддержки v0.0.13:** Если `mode` опущен, инструмент откатывается к поведению v0.0.12, когда `modelRoles` пуст (автоматически записывает значения по умолчанию) с предупреждением через `console.warn`. Если `modelRoles` уже существует, ведёт себя как `mode: "show"`. Этот мост будет удалён в v0.0.14.

### Пользовательские агенты

Поместите `.md`-файл в `~/.pi/agent/agents/` (уровень пользователя) или `.pi/agents/` (уровень проекта, закоммитьте), чтобы добавить своего агента:

```markdown
---
name: my-linter

description: Run ESLint and report violations

tools: read, bash

model: "{{fast}}"

thinking: off
---

You are a linting agent. Run `npx eslint --format json` on the
provided files. Report violations grouped by file. No fixes.
```

Затем ссылайтесь на него в любой фазе: `{ "agent": "my-linter", "task": "Lint src/" }`.

## 🍽️ Мы едим свой собственный корм

Каждая функция в `pi-taskflow` поставляется **через `pi-taskflow`.**

Наш flow `self-improve` — это 10-фазный DAG: он аудирует кодовую базу, исправляет дефекты, проверяет корректность, шлюзует по качеству и выводит отчёт — всё декларативно. Он сохранён как `/tf:self-improve` и запускается перед каждым релизом. Ни один другой оркестратор агентов в экосистеме Pi не строит себя с помощью себя.

| Кампания | Масштаб | Фазы | Результат |
|----------|---------|------|-----------|
| [v0.0.8 dogfood](./docs/dogfooding-v0.0.8-report.md) | Полный аудит кодовой базы → триаж → исправление → проверка | 10 фаз, 234 теста | 13 исправлений, всё проходит |
| [v0.0.6 self-audit](./docs/self-audit-report.md) | inventory → map audit → gate → approval → map fix → reduce | 9 фаз | 11 критических дефектов исправлено |
| [Dogfood межзапускного кеша](./docs/rfc-cross-run-memoization.md) | Реальный рантайм + хранилище на диске | Выделенный тестовый стенд | Корректность кеша при adversarial отпечатках |
| [Adversarial cross-review](./docs/brainstorm-adversarial-review-report.md) | Многоагентная adversarial рецензия | `tournament` + `gate` | Исправление P0 в ключе кеша |
| [Init redesign review](./docs/issue-necessity-review-report.md) | Аудит необходимости → параллельные проверки → вердикт | 7 фаз | Полный план редизайна утверждён |

> **Мета:** мы использовали разветвление `map`, вердикты `gate`, утверждение `approval` (человек в цикле), турнир `tournament` (best-of-N), цикл `loop` (до завершения) и межзапускной кеш `cross-run` из `pi-taskflow` — чтобы построить `pi-taskflow`.

## Статус и ограничения

**v0.0.13** — цикл-до-завершения (фаза `loop`: итерация до условия, сходимости или лимита), турнир (best-of-N с судьёй), межзапускная мемоизация (кеш с адресацией по содержимому с git/файловыми/glob/окружающими отпечатками и TTL), интерактивный `/tf init` с выборщиками моделей с учётом ролей + предпросмотр diff + атомарная запись-слияние, 18 встроенных агентов с 6 ролями моделей. Полный слой управления потоком и надёжности (охранители `when`, `join: any`, `retry`/задержка, `approval`, композиция `flow`, лимиты `budget`, сторожевой таймер бездействия) поверх DSL + DAG-рантайма (`agent`/`parallel`/`map`/`gate`/`reduce`). Встроенные и сохранённые flow, возобновление между сессиями, прогресс в реальном времени и изолированный контекст. Запуск выполняется как один потоковый вызов инструмента.

Известные границы (отслеживаются, ограничены — никаких сюрпризов в середине потока):

- **Нет фонового выполнения без привязки к сессии.** Для запуска требуется открытая сессия Pi. Настоящее фоновое выполнение (и триггеры по событиям/расписанию поверх него) в дорожной карте.
- **Нет `output: "file"`.** Вывод — только текст/JSON. Записывайте файлы через вызов инструмента `write` у агента.
- **`map` требует JSON-массив.** Поле `over` должно разрешаться в массив `{steps.ID.json}`. Оберните текстовый список в одноагентную фазу с `output: "json"`.
- **DAG должен быть ациклическим.** Циклы отклоняются на этапе валидации.
