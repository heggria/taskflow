# 上下文隔离

<Callout type="info">
  使用原始子代理时，每个 transcript 都会涌入你的上下文窗口。使用 taskflow 时，中间结果保留在运行时中。
</Callout>

使用原始子代理时，每个 transcript 都会涌入你的上下文窗口。使用 taskflow 时，中间结果保留在运行时中。

## 问题

一个典型的多步骤智能体工作如下：

1. 探索代码库。
2. 审计 20 个文件。
3. 总结发现。
4. 编写补丁。
5. 审查补丁。

如果每个步骤产生 4K token 的 transcript，你在看到结果之前上下文就增长了 20K token。重复几次后，你会达到限制、丢失早期上下文或为重复工作付费。

## taskflow 模型

在 taskflow 中，每个阶段都在隔离的子代理进程中运行。运行时收集输出，但**在最终阶段完成之前不会将它们推入你的对话**。

```
你的上下文          运行时
├─ 最终输出  ←──  ├─ 阶段 A 输出
                  ├─ 阶段 B 输出
                  ├─ 阶段 C 输出
                  └─ 阶段 D 输出（final）
```

这意味着你可以：

- 并行运行 50 个子代理而不增长上下文。
- 恢复长时间运行而无需重放每个 transcript。
- 将注意力集中在结果上，而不是过程。

## 何时需要窥视

有时你需要检查中间输出进行调试。使用 CLI：

```bash title="窥视中间输出"
/tf peek <runId> <phaseId>
```

这是只读且人工调用的——它不会破坏正常运行时的隔离契约。

## 共享上下文树

默认情况下，子代理完全隔离。对于需要共享昂贵上下文的阶段，通过 `shareContext: true`（或 flow 级别的 `contextSharing: true`）选择加入。

这会给子代理四个工具：

- `ctx_write(key, value)` — 发布发现。
- `ctx_read(key?)` — 读取已发布发现。
- `ctx_report(summary, structured?)` — 向父级报告。
- `ctx_spawn(assignments[])` — 委托子工作。

所有内容都是选择加入、大小受限、深度受限，并随运行清理。
