# DAG 模型

每个 taskflow 都是一个 **DAG** —— 由离散 **任务** 节点组成的有向无环图。

## 为什么是图？

大多数编排器让模型*编写*工作：命令式代码一步步流动，图隐藏在 `if`/`for`/`await` 中。计划只有在代码运行时才存在。

`taskflow` 相反。你**声明**工作为由 `dependsOn` 边连接的命名任务节点图。运行时*在花费任何 token 之前验证该图*。

## 节点和边

**节点**是一个阶段：具有唯一 `id` 和 `type` 的单个工作单元。

**边**是一个 `dependsOn` 数组：它表示"此阶段等待这些其他阶段完成后才开始"。

```json
{
  "phases": [
    { "id": "a", "type": "agent", "task": "步骤 A" },
    { "id": "b", "type": "agent", "task": "步骤 B", "dependsOn": ["a"] },
    { "id": "c", "type": "agent", "task": "步骤 C", "dependsOn": ["a"] },
    { "id": "d", "type": "agent", "task": "步骤 D", "dependsOn": ["b", "c"] }
  ]
}
```

这定义了：

```
a
├─ b ─┐
└─ c ─┤
      d
```

- `a` 先运行。
- `b` 和 `c` 在 `a` 之后并行运行。
- `d` 在 `b` 和 `c` 都完成后运行。

## 依赖方向

依赖是**声明的**，不是推断的。运行时不在乎 `phases[]` 数组中的顺序——它只在乎 `dependsOn`。

这是新用户最常见的错误：按顺序列出但没有 `dependsOn` 的四个阶段是四个**并行**阶段，而不是一条链。

## 无循环

DAG 必须无环。如果阶段 `a` 依赖 `b`，而 `b` 又依赖 `a`，运行时会在验证时拒绝该定义。

## 执行层

运行时按拓扑层执行 DAG：

1. 找到所有无未满足依赖的阶段。
2. 运行它们（遵守 `concurrency` 限制）。
3. 阶段完成时，将其依赖者标记为就绪。
4. 重复直到所有阶段完成、失败或被跳过。

这与 Make、Bazel 或 Vite 等构建系统的执行模型相同。

## 合并语义

默认情况下，阶段等待**所有**依赖（`join: "all"`）。你可以改为**任一**（`join: "any"`），使阶段在*任一*依赖完成时运行。

`join: "any"` 对 if/else 路由至关重要：分支 `deep` *或* 分支 `quick` 运行，合并阶段在任一完成时触发。

## 动态边

静态 `dependsOn` 足以满足大多数 flow。对于运行时才发现的工作，`flow` 阶段可以执行一个由上游阶段生成的子流程。生成的子流程在运行前会验证循环、悬空引用、重复 id 等。
