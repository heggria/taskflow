# 流程定义

**流程定义**是一个 JSON 对象，声明一个多阶段工作流：其名称、调用参数、执行调参、成本上限，以及通过 `dependsOn` 构成 DAG 的有序阶段数组。每个 taskflow —— 无论保存到磁盘还是以简写内联传入 —— 都会被反糖化为 `TaskflowSchema` 验证的相同结构。

<Callout type="info">
  结构上只有 `phases` 是必需的（保存的 flow 还需 `name`）。其他一切都是行为调参。简写规范（`task` / `tasks` / `chain`）完全省略 `phases`，在验证前被反糖化为完整 DAG。
</Callout>

flow 在执行前由 `/tf verify` 以零 token 成本验证。非法 flow 永远不会生成子代理。

## 顶层字段参考

| 字段 | 类型 | 默认值 | 必需 | 说明 |
|---|---|---|---|---|
| `name` | `string`（`minLength: 1`） | — | **是**（保存的 flow） | 工作流标识符。保存后成为 `/tf:<name>` 命令，也是运行记录目录名。必须非空。 |
| `description` | `string` | — | 否 | 人类可读摘要。出现在 `/tf list` 和运行记录中。 |
| `version` | `number` | `1` | 否 | Schema/flow 版本标签，用于作者追踪。运行时不强制。 |
| `args` | `Record<string, ArgSpec>` | — | 否 | 带默认值的已声明调用参数。见 [`args`](#args)。 |
| `concurrency` | `number` | `8` | 否 | 默认最大并发子代理数。被未设自身 `concurrency` 的 `map` / `parallel` 阶段继承。 |
| `budget` | `Budget` | — | 否 | 运行级成本/token 上限。见 [`budget`](#budget)。 |
| `agentScope` | `"user" \| "project" \| "both"` | `"user"` | 否 | 发现 agent 的位置。见 [`agentScope`](#agentscope)。 |
| `strictInterpolation` | `boolean` | `false` | 否 | 将验证警告（运行时缺失 `{args.X}`、cwd/codebase 不匹配、被忽略字段）提升为硬错误。见[验证行为](#验证行为)。 |
| `contextSharing` | `boolean` | `false` | 否 | 为**所有**阶段启用共享上下文树（等价于每个阶段都设 `shareContext: true`）。个别阶段仍可退出。 |
| `incremental` | `boolean` | `false` | 否 | 将每个阶段默认为 `cache.scope: "cross-run"`，使跨会话重跑复用未变阶段。见 [`incremental`](#incremental)。 |
| `phases` | `Phase[]`（`minItems: 1`） | — | **是** | 有序阶段定义。DAG 由每个阶段的 `dependsOn` 构成。见[阶段类型](/zh-cn/docs/syntax/phase-types)。 |

### `ArgSpec`

`args` 中的每个条目是含以下字段的对象：

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `default` | `unknown` | — | 运行时未提供该参数时使用的值。类型不被 schema 强制 —— 插值需谨慎。 |
| `description` | `string` | — | 人类可读的参数文档。 |
| `required` | `boolean` | — | 作者声明的必需标志。（运行时在缺失时应用 `default`；`required` 是建议性元数据。） |

---

## `name`

**保存的 flow 必需。** 非空字符串（`minLength: 1`），用于 `/tf run <name>`、`/tf:<name>` 命令别名和运行状态目录。省略 `name` 的内联/简写 flow 会得到合成的回退名（`"task"`、`"parallel"` 或 `"chain"`）。

```json title="name-基础.json"
{
  "name": "audit-auth",
  "phases": [/* ... */]
}
```

<Callout type="warn">
  缺失或空的 `name` 是硬验证错误：`Missing or invalid 'name'`。这在任何阶段验证之前检查。
</Callout>

---

## `description`

可选的人类可读字符串。不被解析或强制 —— 纯粹用于 `/tf list`、运行记录和文档。

---

## `args`

已声明的调用参数。每个参数可带 `default`、`description` 和建议性 `required` 标志。运行时，`resolveArgs` 将已声明默认值应用到提供的值上，然后透传任何额外提供的键（因此调用者也可提供未声明的参数）。

```json title="args-带默认值.json"
{
  "name": "audit",
  "args": {
    "dir": { "default": "src/routes", "description": "要审计的目录" },
    "force": { "default": false },
    "severity": { "default": "high", "required": true }
  },
  "phases": [/* ... */]
}
```

在调用时传参：

```bash title="带参数运行"
/tf run audit dir=src/api force=true
```

在任意阶段用 `{args.<name>}` 插值参数：

```json title="args-插值.json"
{
  "id": "scan",
  "type": "agent",
  "task": "扫描 {args.dir} 中缺失的认证检查。"
}
```

### 验证行为

<Callout type="tip">
  声明 `args` 是可选的。运行时传入的未声明参数仍可通过 `{args.X}` 访问 —— 只是没有默认值。声明它们可获得默认值和文档。
</Callout>

当 flow **带**已解析调用参数验证时（即在运行启动时，而非静态 `/tf verify`），一个没有匹配提供值**且**没有默认值的 `{args.X}` 引用会产生警告：

> `Taskflow references {args.X} but the invocation did not provide 'X'. The placeholder will remain literal unless a default or runtime arg is supplied.`

在 `strictInterpolation: true` 下，此警告变为硬错误。

---

## `concurrency`

整个 flow 的默认最大并发子代理数。被未设阶段级 `concurrency` 的 `map` 和 `parallel` 阶段继承。默认 `8`。

```json title="concurrency-flow.json"
{
  "name": "fanout",
  "concurrency": 16,
  "phases": [
    { "id": "scan", "type": "map", "over": "{steps.discover.json}", "task": "...", "concurrency": 4 }
  ]
}
```

上面阶段级 `concurrency: 4` 仅对该阶段覆盖 flow 级的 `16`。

---

## `agentScope`

控制运行时在何处发现 agent 定义（带 YAML frontmatter 的 `.md` 文件）。

| 取值 | 搜索路径 | 冲突规则 |
|---|---|---|
| `"user"`（默认） | `~/.pi/agent/agents/*.md` | — |
| `"project"` | `.pi/agents/*.md` | — |
| `"both"` | user，然后 project | 名称冲突时**project 优先** |

```json title="agentscope-project.json"
{
  "name": "team-flow",
  "agentScope": "both",
  "phases": [/* ... */]
}
```

---

## `budget`

运行级成本上限。`Budget` 对象含两个可选上限：

| 字段 | 类型 | 说明 |
|---|---|---|
| `maxUSD` | `number` | 累计成本超过此美元数时停止运行。 |
| `maxTokens` | `number` | 累计输入+输出 token 超过此值时停止运行。 |

```json title="budget-定义.json"
{
  "name": "capped-run",
  "budget": {
    "maxUSD": 2.0,
    "maxTokens": 2000000
  },
  "phases": [/* ... */]
}
```

当任一上限被超出时，剩余阶段被**跳过**，运行以 `blocked` 状态结束。已运行的子代理不会在调用中途被中断；上限在阶段之间检查。

<Callout type="info">
  两个字段都可选。设置其一或两者。`0` 的预算被视为"立即超出" —— 仅对干结构检查有用。
</Callout>

---

## `strictInterpolation`

一个布尔值（默认 `false`），控制验证如何处理非致命问题。

<Callout type="warn">
  `strictInterpolation` **不**控制硬性 `{steps.X}` 可达性检查 —— 那始终是硬错误。它只把*警告*提升为错误。
</Callout>

### 始终为硬错误的部分（与该标志无关）

- `{steps.X.*}` 引用中 `X` 无法经 `dependsOn` / `from`（传递地）到达。该阶段会与其生产者并行运行并看到字面占位符。
- 阶段通过 `{steps.<self>.*}` 引用自身输出（`loop` 阶段除外，它在 `until` 中合法地这样做）。
- 未知的 `dependsOn` / `from` 目标。
- 依赖环。
- 重复的阶段 id。
- 多于一个 `final: true` 阶段。

### 仅在 `strictInterpolation: true` 下变为硬错误的部分

| 警告 | 默认 | 在 `strict` 下 |
|---|---|---|
| 引用了 `{args.X}` 但运行时未提供（且无默认值） | 警告 | **错误** |
| 调用 `cwd` 与 `args.codebase` 不匹配 | 警告 | **错误** |
| 仅 loop 适用的字段（`until`、`maxIterations`、`convergence`）设在非 loop 阶段 | 警告（忽略） | **错误** |
| `idempotent: false` 在 `approval` / `flow` 上是空操作 | 警告 | **错误** |
| `idempotent: false` 覆盖 `cache.scope: "cross-run"` | 警告 | **错误** |
| gate 上同时设 `eval` 和 `score` | 警告 | **错误** |

```json title="strict-开启.json"
{
  "name": "production-flow",
  "strictInterpolation": true,
  "phases": [/* ... */]
}
```

<Callout type="tip">
  推荐用于已保存/生产 flow：它在运行浪费 token 之前捕获静默占位符泄漏和过时字段引用。
</Callout>

---

## `contextSharing`

一个布尔值（默认 `false`），将 flow 中的**每个**阶段纳入共享上下文树。这等价于在每个阶段单独设 `shareContext: true` —— 阶段仍可通过设 `shareContext: false` 退出。

启用后，每个阶段的子代理获得 `ctx_read` / `ctx_write`（与兄弟和祖先共享的 blackboard，避免重复读文件）以及 `ctx_report` / `ctx_spawn`（向上报告 + 排队运行时拾取的子任务）。

<Callout type="warn">
  当上下文共享激活时（经此标志或阶段级 `shareContext`），阶段 id 成为文件系统节点 id，必须匹配 `[A-Za-z0-9._-]+`。两个清理后相同的 id 会静默合并其 blackboard —— 这作为硬错误强制执行。
</Callout>

```json title="contextsharing-开启.json"
{
  "name": "collab-flow",
  "contextSharing": true,
  "phases": [
    { "id": "explore", "type": "agent", "task": "..." },
    { "id": "synthesize", "type": "agent", "task": "...", "dependsOn": ["explore"] }
  ]
}
```

---

## `incremental`

一个布尔值（默认 `false`），将每个阶段默认为跨运行缓存（`cache.scope: "cross-run"`），使重跑 flow 时复用跨运行和跨会话未变的阶段结果。

### 如何与阶段级设置交互

- 等价于在每个阶段设 `cache: { scope: "cross-run" }`。
- 阶段自身的 `cache` 设置**优先** —— `cache.scope: "off"` 或 `"run-only"` 仍被尊重。
- 跨运行被排除的阶段类型（`gate`、`approval`、`loop`、`tournament`、`script`）即使 `incremental` 也**永不**跨运行缓存 —— 它们每次运行必须产生新结果。
- `idempotent: false` 阶段完全排除在缓存之外（它们总是重跑）。在 `incremental` 下，这会产生警告，让作者知道该阶段不会被复用。

<Callout type="info">
  运行时 `incremental` 参数（调用时传入）覆盖 flow 声明的值，让你无需编辑 flow 即可强制全新或增量运行。
</Callout>

```json title="incremental-开启.json"
{
  "name": "ci-audit",
  "incremental": true,
  "phases": [
    { "id": "lint", "type": "script", "run": "npm run lint" },
    { "id": "review", "type": "agent", "task": "审查最近的变更。" }
  ]
}
```

上面 `lint` 是 `script` 阶段 —— 即使 `incremental` 它每次调用都重跑。`review` 以其输入为键跨运行缓存。

---

## `phases`

必需的阶段定义数组（`minItems: 1`）。每个阶段需要唯一 `id` 并接受 `type`（默认 `"agent"`）。阶段通过 `dependsOn` 构成 DAG；无依赖的阶段在第一拓扑层运行。

<Callout type="info">
  若没有阶段标记 `final: true`，数组顺序中**最后**一个阶段成为工作流结果。若多于一个标记 `final`，验证失败。
</Callout>

每个阶段类型的每个字段见[阶段类型参考](/zh-cn/docs/syntax/phase-types)。

### 拓扑执行

运行时通过 Kahn 算法计算拓扑层：同层阶段并发运行（至多 `concurrency` 个）。一个阶段在其所有 `dependsOn` 完成后启动（或在 `join: "any"` 下任一完成后）。

---

## 简写

对一次性委派，可跳过完整 `phases` DAG 使用简写规范。运行时的 `desugar()` 在验证前将其变换为完整 `Taskflow`。当规范**没有** `phases` 数组**但**有 `task`、`tasks` 或 `chain` 之一时，被识别为简写。

### `task` — 单个 agent

```json title="task-简写.json"
{ "task": "总结 src/" }
```

反糖化为：

```json title="task-反糖化.json"
{
  "name": "task",
  "phases": [
    { "id": "main", "type": "agent", "task": "总结 src/", "final": true }
  ]
}
```

可选的 `agent`、`context` 和 `contextLimit` 会透传到阶段。

### `tasks` — 并行 fan-out

```json title="tasks-简写.json"
{
  "tasks": [
    { "task": "审计认证", "agent": "analyst" },
    { "task": "审计校验", "agent": "analyst" }
  ]
}
```

反糖化为单个 `parallel` 阶段（id `"parallel"`，`final: true`）。上下文在所有分支间**共享**：规范级 `context` 加上步骤级 `context` 数组的并集被合并并读取一次。

```json title="tasks-反糖化.json"
{
  "name": "parallel",
  "phases": [
    {
      "id": "parallel",
      "type": "parallel",
      "branches": [
        { "task": "审计认证", "agent": "analyst" },
        { "task": "审计校验", "agent": "analyst" }
      ],
      "final": true
    }
  ]
}
```

### `chain` — 顺序 agent

```json title="chain-简写.json"
{
  "chain": [
    { "task": "找出所有 TODO 注释" },
    { "task": "总结它们：{previous.output}" },
    { "task": "为以下内容开 issue：{previous.output}" }
  ]
}
```

反糖化为顺序 `agent` 阶段（`step1`、`step2`、…），每个依赖前一个；最后一个是 `final: true`。每步用 `{previous.output}` 引用上一步输出。

```json title="chain-反糖化.json"
{
  "name": "chain",
  "phases": [
    { "id": "step1", "type": "agent", "task": "找出所有 TODO 注释" },
    { "id": "step2", "type": "agent", "task": "总结它们：{previous.output}", "dependsOn": ["step1"] },
    { "id": "step3", "type": "agent", "task": "为以下内容开 issue：{previous.output}", "dependsOn": ["step2"], "final": true }
  ]
}
```

<Callout type="warn">
  在 `chain` 模式下，顶层 `context` / `contextLimit` 被**忽略**（运行时会警告）。把它们放在个别步骤上 —— chain 模式下的 `context` 会是 flow 级默认，这是有意不支持的。
</Callout>

### 透传的元数据

所有三种简写形式都把以下可选顶层字段透传到反糖化后的 flow：

- `name`（默认为 `"task"`、`"parallel"` 或 `"chain"`）
- `description`
- `concurrency`
- `agentScope`
- `args`
- `budget`
- `strictInterpolation`

### `isShorthand` 识别

当规范没有 `phases` 数组**且**至少满足以下之一时为简写：

- `chain` 是非空数组
- `tasks` 是非空数组
- `task` 是字符串

否则被视为完整 flow 并按 `TaskflowSchema` 验证。

## 验证行为

每个 flow 在执行前都会验证。验证分两种模式：

1. **静态**（`/tf verify`，无运行时参数）：仅结构检查 —— schema 形状、DAG 完整性、环检测、`{steps.X}` 引用可达性。零 token。
2. **运行时**（运行启动时，带已解析参数）：以上全部加参数存在警告和 cwd/codebase 不匹配警告。

### 硬错误（始终令运行失败）

| 错误 | 原因 |
|---|---|
| `Missing or invalid 'name'` | `name` 缺失、为空或非字符串。 |
| `Taskflow must have at least one phase` | `phases` 缺失或为空。 |
| `Duplicate phase id: X` | 两个阶段共用一个 `id`。 |
| `Phase 'X': unknown type 'Y'` | `type` 不在 10 个允许值中。 |
| `Phase 'X': dependsOn unknown phase 'Y'` | 某个 `dependsOn` 或 `from` 条目没有匹配的阶段。 |
| `Dependency cycle detected: A -> B -> A` | Kahn 算法发现环。 |
| `Only one phase may be marked 'final' (found N)` | 多于一个 `final: true` 阶段。 |
| `Phase 'X': task references {steps.Y.*} but 'Y' is not reachable via dependsOn` | `{steps.Y}` 引用中 `Y` 不是传递祖先（除非 `join: "any"`）。 |
| `Phase 'X': references its own output via {steps.X.*}` | 自引用（`loop` 阶段除外）。 |
| `Phase 'X': id uses underscores` | 阶段 id 含 `_`（用连字符）。 |
| `Phase 'X': agent name 'Y' uses underscores` | agent 名含 `_`。 |
| `Phase 'X': agent 'Y' has invalid name format` | agent 不匹配 `/^[a-z][a-z0-9-]*$/`。 |
| `Phase 'X': ids used with context sharing must match [A-Za-z0-9._-]+` | 上下文共享阶段 id 字符集非法。 |

### 警告（非致命；在 `strictInterpolation` 下提升为错误）

| 警告 | 说明 |
|---|---|
| 引用了 `{args.X}` 但未提供 | 无默认值，无运行时值。占位符保持字面。 |
| 调用 `cwd` ≠ `args.codebase` | 依赖 cwd 的 agent 可能检查错误的仓库。 |
| 仅 loop 字段在非 loop 阶段 | `until` / `maxIterations` / `convergence` 被静默忽略。 |
| `idempotent: false` 在 `approval` / `flow` 上是空操作 | 这些阶段不运行子代理（无瞬态重试），且已排除在跨运行缓存之外。该标志无效。 |
| `idempotent: false` 覆盖 `cache.scope: "cross-run"` | 有副作用的阶段永不缓存。 |
| `incremental` 下的 `idempotent: false` | 阶段排除在缓存之外，每次调用都重跑。 |
| gate 上同时设 `eval` 和 `score` | `eval` 先运行；通常一个就够。 |

<Callout type="tip">
  尽早并经常运行 `/tf verify`。它以零成本在任何子代理生成前捕获上述每个硬错误。
</Callout>

## 完整示例

```json title="flow-定义-完整.json"
{
  "name": "audit-auth",
  "description": "审计路由是否缺失认证检查",
  "version": 2,
  "args": {
    "dir": { "default": "src/routes", "description": "要审计的目录" },
    "severity": { "default": "high", "required": true }
  },
  "concurrency": 8,
  "agentScope": "both",
  "contextSharing": false,
  "strictInterpolation": true,
  "incremental": false,
  "budget": {
    "maxUSD": 2.0,
    "maxTokens": 2000000
  },
  "phases": [
    {
      "id": "discover",
      "type": "agent",
      "agent": "scout",
      "task": "以 JSON 列出 {args.dir} 中的每个路由处理器。",
      "output": "json"
    },
    {
      "id": "audit-each",
      "type": "map",
      "over": "{steps.discover.json}",
      "as": "route",
      "agent": "security-reviewer",
      "task": "{route.path} 是否需要认证？输出 {\"ok\": true|false}。",
      "output": "json",
      "dependsOn": ["discover"]
    },
    {
      "id": "report",
      "type": "reduce",
      "from": ["audit-each"],
      "agent": "writer",
```
