# The DAG Model

Every taskflow is a **DAG** — a directed acyclic graph — of discrete **task** nodes.

## Why a graph?

Most orchestrators let the model *script* the work: imperative code that flows step by step, with the graph hidden inside `if`/`for`/`await`. The plan only exists as the code runs.

`taskflow` does the opposite. You **declare** the work as a graph of named task nodes connected by `dependsOn` edges. The runtime *verifies that graph before it spends a single token*.

## Nodes and edges

A **node** is a phase: a single unit of work with a unique `id` and a `type`.

An **edge** is a `dependsOn` array: it says "this phase waits for these other phases to finish before it starts."

```json title="a-b-c-d.json"
{
  "phases": [
    { "id": "a", "type": "agent", "task": "Step A" },
    { "id": "b", "type": "agent", "task": "Step B", "dependsOn": ["a"] },
    { "id": "c", "type": "agent", "task": "Step C", "dependsOn": ["a"] },
    { "id": "d", "type": "agent", "task": "Step D", "dependsOn": ["b", "c"] }
  ]
}
```

This defines:

```
a
├─ b ─┐
└─ c ─┤
      d
```

- `a` runs first.
- `b` and `c` run in parallel after `a`.
- `d` runs after both `b` and `c` complete.

## Dependency direction

Dependencies are **declared**, not inferred. The runtime does not care about the order of phases in the `phases[]` array — it cares about `dependsOn`.

<Callout type="warn">
  This is the most common mistake for new users: four phases listed in order with no `dependsOn` are four **parallel** phases, not a chain.
</Callout>

## No cycles

A DAG must be acyclic. If phase `a` depends on `b` and `b` depends on `a`, the runtime rejects the definition at verification time.

## Execution layers

The runtime executes the DAG in topological layers:

1. Find all phases with no unmet dependencies.
2. Run them (respecting `concurrency` limits).
3. When a phase finishes, mark its dependents as ready.
4. Repeat until all phases complete, fail, or are skipped.

This is the same execution model as build systems like Make, Bazel, or Vite.

## Join semantics

By default, a phase waits for **all** its dependencies (`join: "all"`). You can change this to **any** (`join: "any"`) so the phase runs as soon as *one* dependency completes.

`join: "any"` is essential for if/else routing: branch `deep` *or* branch `quick` runs, and the merge phase fires when either finishes.

## Dynamic edges

Static `dependsOn` is enough for most flows. For runtime-discovered work, a `flow` phase can execute a sub-flow whose shape is generated by an upstream phase. The generated sub-flow is validated (cycles, dangling refs, duplicate ids) before it runs.
