# What is taskflow?

<Callout type="info">
  **taskflow** is a declarative, verifiable *graph of tasks* for coding-agent subagents.
</Callout>

In plain English: it lets you describe multi-step agent work as a **DAG** (directed acyclic graph) of discrete task nodes, validates that graph before any model call runs, and then executes it with dynamic fan-out, gates, loops, tournaments, and cross-session resume.

## Why "taskflow" and not "workflow"?

The name is the thesis.

In engineering, a **task** is a *discrete, declared unit of work* — the node of a task graph (the same `task` a build system, scheduler, or compiler wires into a DAG). **Work**, by contrast, is *fluid and unbounded* — the continuous, imperative act of doing.

That distinction is exactly the design split playing out across coding-agent ecosystems:

- A **`workflow`** (the dynamic, code-mode kind) is the model writing an **imperative script** that *flows*: `await agent(...)`, an `if`, a `for`, another `await`. Expressive — it's Turing-complete — but the graph only exists *as the code runs*. You can't see it, diff it, or prove it terminates before you pay for it.
- A **`taskflow`** moves the plan **out of code and into a declarative graph of `task` nodes.** Because the graph is *data*, the runtime can do what an imperative script structurally cannot: **statically verify it** (no cycles, no dead ends, no budget overflow, no dangling refs) before a single subagent spawns, **render it**, **resume it** phase-by-phase, and **save it** as a one-word command.

> **The trade we make on purpose:** we give up the raw expressivity of arbitrary code to gain something an imperative script can't have — a graph that is **verifiable, observable, replayable, and safe to generate with an LLM.**

## Core ideas

### 1. The plan is data

A taskflow definition is JSON (or YAML, or MDX frontmatter). It declares:

- **Phases** — discrete task nodes (`agent`, `map`, `gate`, `reduce`, `loop`, `tournament`, `approval`, `flow`, `script`).
- **Edges** — `dependsOn` relationships that form the DAG.
- **Control flow** — `when` guards, `join: any/all`, `retry`, `budget`.

Because it is data, the runtime can verify it, visualize it, and cache its results.

### 2. Static verification before tokens

Before executing, the runtime checks:

- No cycles in the DAG.
- No dead-end phases.
- No dangling `dependsOn` references.
- No unresolved interpolation placeholders (with `strictInterpolation`).
- No budget overflow.

This is the main reason to prefer a declarative graph over an imperative script: you can *prove* structural properties of the plan before spending money on it.

### 3. Context isolation

With raw subagents, every transcript floods your context. With taskflow, every intermediate output is held in the runtime. Only the final phase output returns to your conversation.

This means you can run 50 subagents in parallel and still keep your context window small.

### 4. Cross-session resume

Every completed phase is written to disk. A run that fails or is stopped can be continued later with `/tf resume <runId>` — cached phases auto-skip, and only the remaining work spends tokens.

## When to use taskflow

Use taskflow when:

- A job has **multiple dependent steps**.
- You need to **fan out** over many items.
- You want a **quality gate** before final output.
- The plan should be **reusable** by name.
- You want **cost control** or **human approval**.
- The plan might be **generated by another model** and must be checked before running.

Don't use taskflow for one-off, single-step delegations — the built-in subagent tool is already perfect for that.

```txt title="The taskflow pipeline"
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Author defines │ ──▶ │ Runtime verifies │ ──▶ │ Runtime executes│
│  taskflow JSON  │     │ DAG before tokens│     │ phase-by-phase  │
└─────────────────┘     └──────────────────┘     └─────────────────┘
                               │                          │
                               ▼                          ▼
                        ┌─────────────┐          ┌─────────────┐

                        │   report    │          │  only       │
                        └─────────────┘          └─────────────┘
```
