### Implementation order

```
Feature C (idempotent) → Feature A (score) → Feature B (reflexion)
```

**Rationale:**
1. **C (idempotent):** Smallest, lowest-risk, highest-ROI for the existing `incremental` feature. 1 cache-scope line + 1 retry line — lands in hours, unlocks safe `incremental` usage immediately.

## Feature specs (as arbitrated)

### [1/3] Feature A — Eval/Scorer Phase Capability

#### Goal

Give `gate` phases a `score` field that runs deterministic scorers (exact-match, contains, regex, json-schema, length-range, code-compiles) and an optional `llm-judge` fallback. Scorers combine via `all`/`any`/`weighted` against a `threshold`. When deterministics pass, the gate auto-passes with **zero LLM tokens** (mirroring the `eval` fast-path); when they fail, control falls through to the judge or the gate's regular `task`. The structured score result is stored as `ps.json` and is reachable downstream via `{steps.X.json.combined}` / `{steps.X.json.results.0.passed}`. No 11th phase type; `score` is a field on `gate`.

#### DSL Design

```jsonc
{
  "id": "quality",
  "type": "gate",
  "dependsOn": ["gen"],
  "score": {
    "target": "{steps.gen.output}",        // optional; interpolation ref. Default: "{previous.output}"
    "scorers": [
      { "type": "exact-match", "value": "42", "name": "answer" },
      { "type": "contains", "value": "PASS", "name": "has-pass" },
      { "type": "regex", "pattern": "\\berror\\b", "negate": true, "name": "no-errors" },
      { "type": "json-schema", "schema": { "type": "object", "required": ["score"], "properties": { "score": { "type": "number" } } }, "name": "shape" },
      { "type": "length-range", "min": 10, "max": 5000, "name": "len" },
      { "type": "code-compiles", "language": "typescript", "name": "compiles" }
    ],
    "combine": "weighted",                 // "all" (default) | "any" | "weighted"
    "weights": [1, 1, 2, 1, 1, 3],         // required iff combine:"weighted"; length must equal scorers.length
    "threshold": 0.8,                      // combine:"weighted" only; default 0.5; must be in (0, 1]
    "judge": {                             // optional LLM-as-judge fallback
      "agent": "reviewer",                 // hyphen-only name; resolved via resolveAgent (falls back to default)
      "task": "Score correctness 0.0-1.0. Return JSON {\"score\": 0.0-1.0, \"verdict\": \"pass\"|\"block\", \"reason\": \"...\"}."  // interpolation supported; runtime appends target + scorer report
    }
  },
  "task": "Optional qualitative review… VERDICT: PASS or BLOCK."  // optional when score.judge present
}
```

| `score.scorers` | `Scorer[]` | — | Required, non-empty. Each `Scorer` is `Type.Object({type, name?, ...}, {additionalProperties:false})` |

| `score.scorers[].name` | `string` | `<type>-<index>` | Used in result JSON; hyphen-only convention |
| `score.scorers[].value` | `string` | — | Required for `exact-match`, `contains` |
| `score.scorers[].pattern` | `string` | — | Required for `regex` (JS RegExp; invalid → scorer fails with detail, treated as not-passed) |
| `score.scorers[].negate` | `boolean` | `false` | `regex` only: invert the match result |

#### Design constraints (verified against source)

- **Extends gate, not a new type**: `score` is a `PhaseSchema` field (PhaseSchema at `schema.ts:115`); `PHASE_TYPES` unchanged (10 types at `schema.ts:18`).
- **Zero new deps**: `node:child_process` (already used by `script` phase at `runtime.ts:1259`) for `code-compiles`; TypeBox for schema; no npm packages.
- **Scorers are pure functions**: the 5 string scorers live in `scorers.ts` (pure, zero-dep). `code-compiles` is impure (spawns a compiler) and lives in `scorer-runtime.ts` — dispatched by the runtime, never imported into the pure module. Mirrors the `contract.ts` (pure at `contract.ts:1-154`) ↔ `runtime.ts` (impure) split.
- **llm-judge reuses `runOne`**: judge runs via the existing `runOne` closure (same path as the gate `task` at `runtime.ts:1196`), with `resolveAgent` fallback (at `runtime.ts:1185`).
- **Fail-open on ambiguity**: per project invariants (documented at AGENTS.md "Critical Invariants") — judge-output parse errors fail-open to PASS; unresolved `target` fails-open to "run the LLM"; deterministic **explicit** failures are NOT ambiguity (they BLOCK when no LLM fallback exists).
- **Structured JSON downstream**: score result stored as `ps.json`; reachable via `{steps.X.json.combined}` etc. (`dig()` at `interpolate.ts:127` handles `results.0.passed`).

#### Runtime semantics

**Evaluation order** (NEW score fast-path inserted at `runtime.ts:1180` — after the `eval` fast-path block ending at `runtime.ts:1180` and before `const interp = interpolate(phase.task...)` at `runtime.ts:1184`):

```
0. (existing) eval[] fast-path at 1119 — unchanged. If eval present and all pass, returns early.
1. NEW: if type==="gate" && phase.score?.scorers?.length > 0 → runScoreGate():
   a. Resolve target = interpolate(phase.score.target ?? "{previous.output}", ctx).text
      - If missing refs (interp.missing.length>0) → fail-open: skip scorers, fall to step 1e.
   b. Run deterministic scorers: for each scorer, dispatch:
        - 5 pure types → evaluatePureScorer(scorer, target)  [scorers.ts]
        - code-compiles → await runCodeCompilesScorer(scorer, target, effCwd)  [scorer-runtime.ts]
      Each → ScorerResult { name, type, passed: boolean, score: 0|1, detail?: string }.
   c. Combine deterministic results per combine:
        - "all":  detPassed = results.every(r=>r.passed);  detCombined = detPassed?1:0
        - "any":  detPassed = results.some(r=>r.passed);  detCombined = detPassed?1:0
        - "weighted": detCombined = Σ(score_i·w_i)/Σw_i;  detPassed = detCombined >= threshold
   d. If detPassed:
        → AUTO-PASS fast-path (mirror eval-skip at 1163):
          inputHash = cacheKeys(cc, [phase.id, "score-skip", JSON.stringify(phase.score)]).key
          ps = { id, status:"done", output:"PASS (scorers passed — no LLM call)",
                 gate:{verdict:"pass"}, json: scoreResultJSON(detResults, detCombined, threshold, "pass"),
                 usage: emptyUsage(), inputHash, endedAt: now }
          recordCache(cc, ps); return ps.
   e. If !detPassed:
        - If score.judge present: judge runs (gets target + scorer report appended), verdict authoritative, fail-open to PASS on unparseable.
        - Else if phase.task present: gate task runs with scorer report appended.
        - Else: deterministic-only BLOCK (explicit failure is not ambiguity — invariant does not apply).
   f. onBlock:retry — refactored to `attemptGate` closure so score and non-score gates route through it.
   g. recordCache(cc, ps); return ps.
```

### [2/3] Feature B — Reflexion Memory in Loops

#### Goal

Add **reflexion memory to `loop` phases**: when a loop phase sets `reflexion: true`, each iteration `i > 1` receives a `{reflexion}` placeholder (or an auto-appended block) carrying a structured failure summary of the **immediately prior iteration** — prior output snippet, failure category, `expect`-contract diagnostics, and the unsatisfied `until` condition. Iteration 1 receives a sentinel. The loop **continues past body failures** (instead of breaking on first failure), since contract failures are the strongest signal source.

#### Central design decision (resolved)

The current loop terminates on the first failed iteration (`runtime.ts:1795` `if (isFailed(r)) { failedResult = r; stop = "failed"; break; }`). For reflexion to be meaningful, **when `reflexion: true`, body failures (subagent error / `expect` contract violation / timeout) do NOT terminate the loop** — they become reflexion context for the next iteration. The loop still terminates on: `until` satisfied, convergence, `maxIterations`, abort, or over-budget. Only when `maxIterations` is exhausted *with* failures does the loop fail (carrying the last failure). When `reflexion` is false/absent, behavior is **byte-for-byte unchanged** — this is the backward-compat invariant and the key acceptance gate.

#### DSL design

- **Field**: `reflexion?: Type.Optional(Type.Boolean())` on `PhaseSchema`, valid **only** on `loop` phases. Default `false`.
- **Placeholder**: `{reflexion}` — resolves to a single string (no sub-path traversal; mirrors `{previous.output}`). Iteration 1 resolves to the sentinel `_(first iteration — no prior feedback yet)_`.
- **Auto-append**: When `reflexion: true`, the task lacks `{reflexion}`, and `i > 1`: auto-append the reflexion block delimited by `\n\n---\n\n`, AND record a one-time warning.
- **Size cap**: `REFLEXION_MAX_CHARS = 2000` (exported from `reflexion.ts`). Re-truncate embedded output snippet to fit.

#### Reflexion summary structure (markdown)

```
## Reflexion: iteration N (prior)
- Outcome: <succeeded-but-until-not-met | failed: contract-violation | failed: subagent-error | failed: timeout>
- Stop condition not met: `until` expression evaluated false   (only when prior succeeded)
- Contract diagnostics:                                       (only on contract failure)
  - $.score: required key is missing
- Error: <errorMessage>                                        (only on subagent/timeout failure)
- Prior output (truncated to <cap> chars):
  ```
  <output snippet>
  ```
```

Section presence is conditional (no empty sections). Output snippet is truncated head+tail if over the cap (keep first ~60% + `…[truncated N chars]…` + last ~30%).

#### Runtime semantics

**Where assembled**: inside the `for (let i = 1; i <= maxIters; i++)` loop at `runtime.ts:1775`, immediately before `const bodyCtx = buildInterpolationContext(...)` at `runtime.ts:1789`. The reflexion string is threaded into `bodyCtx` via a new `reflexion` field on `InterpolationContext` (`interpolate.ts:20-33`).

**Iteration cycle (reflexion on):**
1. If `i > 1`: build `reflexionSummary` from immediately prior iteration's outcome. Wrap in try/catch — on throw, `reflexionSummary = undefined` and push a warning (fail-open).
2. Build `bodyCtx` with `reflexion: reflexionSummary` (sentinel for `i === 1`).
3. Interpolate `phase.task`. If reflexion on, task lacks `{reflexion}`, and `i > 1`: auto-append; record one-time warning.
4. `runOne(...)` as today.
5. **Outcome capture** (new):
   - If `isFailed(r)`:
     - When `reflexion` is **off**: set `failedResult`, `stop="failed"`, break (unchanged).
     - When `reflexion` is **on**: record `lastFailedResult = r`, `lastIterationKind = "failure"`, `failedThisLoop = true`; **do not break**; continue. Exception: timeout and abort STILL terminate (consistent with "timedOut never retried" invariant).
   - If success: clear `lastFailedResult`, set `lastIterationKind = "success"`, update outputs.
6. `until`/convergence checks run only on successful iterations.

**After the loop**: if `maxIterations` was hit AND the last iteration was a failure, return `status:"failed"` with the last failure's error. Reflexion defers failure, it doesn't erase it.

#### Interaction with convergence, until, cache

- **Convergence**: compares successful outputs only. Reflexion changes the prompt, not the convergence check.
- **`until`**: evaluated only on successful iterations. Reflexion feeds the prior iteration into the next prompt. No conflict.
- **Cache key**: `firstBodyCtx` includes `reflexion: REFLEXION_SENTINEL` so the key reflects the true first prompt (avoids baking literal `{reflexion}` into the hash).

### [3/3] Feature C — Side-Effect Classification (`idempotent: false`)

#### Goal

Add an `idempotent` field to `PhaseSchema` (boolean, default `true`) that marks phases with irreversible side effects. When `false`: (1) **transient-error auto-retry is suppressed** (only explicit `retry{}` is honored), (2) **all caching is disabled** (within-run resume, cross-run serve and store, map per-item caching). The flag introduces no new execution paths — it gates three existing mechanisms via two single-line hooks.

#### DSL design

```ts
idempotent: Type.Optional(Type.Boolean({
  description:
    "Marks whether this phase is safe to auto-retry and cache. Defaults to true " +
    "(safe). Set to false for phases with irreversible side effects (webhook POSTs, " +
    "file mutations, DB writes): transient provider errors are NOT auto-retried, " +
    "and the result is never served from or written to any cache (within-run resume " +
    "or cross-run). Explicit `retry{}` is still honored — it is the author's declaration " +
    "that a repeat is acceptable. The phase state records `sideEffect: true` for audit.",
})),
```

**Default `true`** → 100% backward compatibility. `phase.idempotent` is `undefined`; `phase.idempotent !== false` is `true` everywhere it's checked.

#### Validation rules

- **Warning** (non-fatal) when `idempotent: false` + `cache.scope: "cross-run"`: the runtime silently downgrades to `"off"`; the author should know.
- **Warning** when `flow.incremental === true` + `idempotent: false` without explicit `cache`: "phase is non-idempotent under an incremental flow — it will re-run every time (this is safe; caching is disabled)."
- **No error** for `idempotent: false` + explicit `retry{}` — that combination is intentional and supported (author declares "this side-effecting op is safe to repeat on failure").
- **Allow on all types** (document low-value ones like `approval`/`gate`).

|-----------|-------------------|---------------------|---------------|
| **Transient-error auto-retry** (3x, `DEFAULT_TRANSIENT_RETRIES=3` at `runtime.ts:862`) | ✅ applied | ❌ suppressed | ~line 947 |

#### Explicit `retry{}` stays honored (justification)

- `idempotent:false` means "the runtime must not *surprise* the author by repeating it." The implicit transient retry (3x backoff on 429/5xx) is a *runtime safety net* the author did not ask for — repeating a side-effecting call behind the author's back is exactly the hazard the flag exists to prevent. **Suppressed.**
- Explicit `retry{max:N}` is the author's *deliberate declaration* that "I accept this operation may be attempted up to N+1 times." **Honored.**
