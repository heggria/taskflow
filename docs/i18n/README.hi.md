<p><strong><a href="https://pi.dev">Pi</a> उप-एजेंटों के लिए घोषणात्मक DAG ऑर्केस्ट्रेशन (declarative DAG orchestration)।</strong><br/>
फैन आउट करें · गेट लगाएँ · पुनर्प्रारंभ करें · कमांड के रूप में सहेजें — मध्यवर्ती परिणाम आपके कॉन्टेक्स्ट से बाहर रहते हैं।</p>

```bash
pi install npm:pi-taskflow
```

</div>

---

**उप-एजेंट (subagents) फायर-एंड-फ़ॉरगेट होते हैं। टास्कफ़्लो फायर करते हैं, फैन आउट करते हैं, रुकते हैं, गेट लगाते हैं, पुनर्प्रारंभ होते हैं, और स्वयं को एक कमांड के रूप में सहेजते हैं।**

आप पहले से ही बिल्ट-इन उप-एजेंट टूल के `task` / `tasks` / `chain` को जानते हैं। `pi-taskflow` उसी *शॉर्टहैंड* का उपयोग करता है — ताकि आपके मौजूदा प्रतिनिधित्व तुरंत **ट्रैक किए जाने योग्य, पुनर्प्रारंभ करने योग्य, और एक-शब्द वाले `/tf:<name>` कमांड के रूप में सहेजने योग्य** बन जाएँ। जब शॉर्टहैंड अपर्याप्त हो, तो पूरा DSL आपको एक वास्तविक DAG देता है: दर्जनों आइटमों पर डायनामिक फैन-आउट, सशर्त रूटिंग, क्वालिटी गेट्स, मानव अनुमोदन, पुनर्प्रयास, और एक कठिन व्यय सीमा (spend ceiling)।

और पूरे समय, **केवल अंतिम चरण (final phase) आपकी बातचीत तक पहुँचता है।** हर मध्यवर्ती ट्रांसक्रिप्ट रनटाइम में रहता है, कभी आपके कॉन्टेक्स्ट विंडो में नहीं।

## यह क्यों मौजूद है

यहाँ वह दीवार है जिससे आप कच्चे उप-एजेंटों (raw subagents) के साथ टकराते हैं: आप गद्य (prose) में एक बहु-चरणीय योजना का वर्णन करते हैं, मॉडल हर बार उसे फिर से व्युत्पन्न करता है, मध्यवर्ती ट्रांसक्रिप्ट आपके कॉन्टेक्स्ट में बाढ़ ला देते हैं, और जैसे ही एक मॉडल कॉल विफल होता है आप शून्य से शुरू करते हैं। कोई पुन: उपयोग नहीं, कोई पुनर्प्राप्ति नहीं, कोई संरचना नहीं।

`pi-taskflow` योजना को **प्रॉम्प्ट से बाहर और एक घोषणात्मक परिभाषा (declarative definition) में ले जाता है।** रनटाइम DAG, लूप्स, पुनर्प्रयासों और मध्यवर्ती अवस्था (intermediate state) का स्वामी होता है। आप एक बार पाइपलाइन घोषित करते हैं और उसे सौ बार — नाम से — चलाते हैं।

> जब किसी कार्य में शाखाकृत फैन-आउट के साथ बारह चरण और एक समीक्षा गेट (review gate) की आवश्यकता होती है, तो आप ऑर्केस्ट्रेशन चाहते हैं — न कि भाग्यशाली प्रॉम्प्टिंग।

| | subagent (बिल्ट-इन) | **pi-taskflow** |
|---|---|---|
| **संचालक (Who drives)** | मॉडल, बारी-बारी से | रनटाइम, एक परिभाषा से |
| **टोपोलॉजी (Topology)** | चेन / सपाट समानांतर | **स्तरित समवर्ती + रूटिंग के साथ DAG** |
| **मध्यवर्ती परिणाम (Intermediate results)** | आपके कॉन्टेक्स्ट विंडो में | **रनटाइम में — आपके कॉन्टेक्स्ट में नहीं** |
| **पैमाना (Scale)** | मुट्ठी भर कार्य | **डायनामिक `map` फैन-आउट दर्जनों आइटमों पर** |
| **पुन: प्रयोज्य (Reusable)** | हर बार पुन: वर्णित | **`/tf:<name>` के रूप में सहेजा गया** |
| **पुनर्प्रारंभ करने योग्य (Resumable)** | ✗ | **✓ क्रॉस-सेशन — कैश्ड चरण अपने-आप छोड़े जाते हैं** |
| **गुणवत्ता गेट (Quality gates)** | ✗ | **`gate` चरण जो `VERDICT: BLOCK` पर रुक जाते हैं** |
| **सशर्त रूटिंग (Conditional routing)** | ✗ | **`when` गार्ड + `join: any` OR-जॉइन** |
| **दोष सहनशीलता (Fault tolerance)** | ✗ | **प्रति-चरण `retry` + क्षणिक त्रुटियों पर ऑटो-रीट्राई** |
| **मानव-इन-द-लूप (Human-in-the-loop)** | ✗ | **`approval` चरण (स्वीकृत / अस्वीकृत / संपादित)** |
| **लागत नियंत्रण (Cost control)** | ✗ | **रन-व्यापी `budget` (USD / टोकन सीमाएँ)** |
| **संरचना (Composition)** | ✗ | **`flow` चरण सहेजे गए उप-फ़्लो चलाते हैं** |
| **लाइव प्रगति (Live progress)** | चलते समय अपारदर्शी | **समय + लागत के साथ लाइव DAG रेंडर** |
| **एर्गोनॉमिक्स (Ergonomics)** | हर बार इनलाइन JSON | **शॉर्टहैंड (`task`/`tasks`/`chain`) *या* DSL** |

यह उप-एजेंट टूल को प्रतिस्थापित नहीं करता। यह आपके उप-एजेंटों को एक DAG, एक स्मृति, और एक नाम देता है।

**कैसे चुनें:**

- **`@pi-agents/orchid`** इकोसिस्टम में सबसे अधिक फ़ीचर-संपूर्ण ऑर्केस्ट्रेटर है (DAG + वर्कट्री + राल्फ़ लूप + एजेंट मेलबॉक्स) — लेकिन इसका DSL एक *निश्चित* 9-चरण पाइपलाइन है, इसमें रनटाइम निर्भरताएँ + jiti हैं, और यह बीटा में है। `pi-taskflow` तब लें जब आप **अपना स्वयं का ग्राफ़ परिभाषित** करना चाहते हैं (न कि किसी निर्धारित को अपनाना) **शून्य निर्भरताओं** और एक-कमांड इंस्टॉल के साथ।
- **`pi-crew` / `ultimate-pi`** भारी जाते हैं — वर्कट्री अलगाव, टिकाऊ एसिंक्रोनस टीमें, बहु-स्तरीय शासन। यदि आप हल्का, घोषणात्मक, और शून्य-निर्भरता चाहते हैं, तो यह परियोजना है।
- **`@zhushanwen/pi-workflow`** भावना में सबसे करीब है और शून्य-निर्भरता वाला भी है, लेकिन आप वर्कफ़्लो को **JavaScript स्क्रिप्ट** के रूप में लिखते हैं। `pi-taskflow` का **घोषणात्मक JSON DSL** अधिक सुरक्षित और ऑडिट करने योग्य है, और इसका **चरण-स्तरीय इनपुट-हैश पुनर्प्रारंभ** कॉल-कैश डेडुप से अधिक ग्रैन्युलर है।
- **`@fiale-plus/pi-rogue-orchestration`** में एक वास्तविक **लूप-अनटिल-डन** (एक सुविधा जो `pi-taskflow` में अभी तक नहीं है) है। यदि आपका कार्य "लक्ष्य पूरा होने तक जारी रखें" है, तो यह देखने लायक है; `pi-taskflow` *संरचित, शाखाकृत* पाइपलाइनों के लिए है।
- **`pi-subagents` / `@gotgenes/pi-subagents`** तदर्थ "इस डिफ़ पर समीक्षक का उपयोग करें" प्रतिनिधित्व और पृष्ठभूमि कार्यों के लिए परिपक्व विकल्प हैं। `pi-taskflow` तब है जब उन प्रतिनिधित्वों को एक *दोहराने योग्य, पुनर्प्रारंभ करने योग्य पाइपलाइन* बनने की आवश्यकता होती है।
- **`pi-pipeline` / `pi-agent-flow`** *निर्धारित, निश्चित* फ़्लो भेजते हैं। `pi-taskflow` एक *खाली कैनवास* भेजता है: आप (या मॉडल) उस कार्य के अनुरूप ग्राफ़ घोषित करते हैं।

> ईमानदार एक-लाइनर: **`pi-taskflow` एकमात्र Pi एक्सटेंशन है जो आपको एक घोषणात्मक, पुनर्प्रारंभ करने योग्य, DAG-आकार की उप-एजेंट पाइपलाइन देता है जिसे आप एक-शब्द वाले कमांड के रूप में सहेजते हैं — शून्य रनटाइम निर्भरताओं और डिज़ाइन द्वारा कॉन्टेक्स्ट पृथक्करण के साथ।** ज्ञात अंतराल जिन्हें यह आगे बंद कर रहा है: लूप-अनटिल-डन, वर्कट्री अलगाव, और नॉन-ब्लॉकिंग पृष्ठभूमि रन (देखें [`STRATEGY.md`](./STRATEGY.md))।

## 30-सेकंड की शुरुआत

**1. इंस्टॉल करें** — एक कमांड:

```bash
pi install npm:pi-taskflow
```

> **वैकल्पिक:** 18 बिल्ट-इन एजेंटों के मॉडल भूमिकाओं (`fast`, `strong`, `thinker`, …) को अपने मॉडलों से मैप करने के लिए एक बार `/tf init` चलाएँ — एक इंटरैक्टिव पिकर।
> इसे छोड़ें और एजेंट बस Pi के डिफ़ॉल्ट मॉडल का उपयोग करेंगे। देखें [मॉडल भूमिकाएँ](#model-roles)।

**2. चलाएँ** — बस Pi सत्र में मॉडल से पूछें:

> *एक चेन चलाएँ: पहले auth फ़्लो का अन्वेषण करें, फिर निष्कर्षों को संक्षेपित करें।*

मॉडल स्वचालित रूप से `taskflow` टूल को कॉल करता है। आपको लाइव प्रगति, प्रति-चरण समय, टोकन लागत, और एक सहेजा गया रन रिकॉर्ड मिलता है — **बिल्ट-इन टूल के समान प्रयास, अब ट्रैक और पुनर्प्रारंभ करने योग्य।**

**3. सहेजें** — कहें *"इसे सहेजें"* और आपके पास हमेशा के लिए `/tf:<name>` होगा।

बस इतना ही। आप अपनी कॉफ़ी के ठंडा होने से पहले अपना पहला वर्कफ़्लो चला सकते हैं — बिना एक भी चरण परिभाषा लिखे।

### शॉर्टहैंड (बिल्ट-इन टूल के समान आकार)

```jsonc
// एकल — एक एजेंट, एक कार्य
{ "task": "src/ की आर्किटेक्चर का सारांश दें", "agent": "explorer" }

// समानांतर — एक साथ कई चलाएँ, आउटपुट मर्ज होते हैं
{ "tasks": [
  { "task": "src/api में auth का ऑडिट करें",             "agent": "analyst" },
  { "task": "src/api में इनपुट वैलिडेशन का ऑडिट करें", "agent": "analyst" }
] }

// चेन — क्रमिक; प्रत्येक चरण पिछले आउटपुट को देखता है
{ "chain": [
  { "task": "src/lib के सार्वजनिक API को सूचीबद्ध करें", "agent": "scout" },
  { "task": "इसके लिए दस्तावेज़ लिखें:\n{previous.output}", "agent": "writer" }
] }
```

`agent` वैकल्पिक है (डिफ़ॉल्ट रूप से पहले खोजे गए एजेंट का उपयोग करता है)। रन को लेबल करने और इसे कमांड के रूप में सहेजने में सक्षम बनाने के लिए एक `name` जोड़ें।

## इसे चलते देखें

यह एक मॉकअप नहीं है। **यह एक वास्तविक रन से stdout है** — `self-improve` फ़्लो जो अपने स्वयं के परीक्षण सूट लिखता और सत्यापित करता है, एक गुणवत्ता गेट द्वारा मध्य-उड़ान में पकड़ा गया:

```
⊗ taskflow self-improve  6/7 · blocked · $0.095
    ✓ discover            agent   deepseek-v4-flash  10t ↑38k ↓6.7k $0.011
  ┌ ✓ write-runner-tests  agent   claude-sonnet-4-6  10t ↑13 ↓6.6k $0.020
  ├ ✓ write-store-tests   agent   claude-sonnet-4-6  10t ↑11 ↓10k $0.018
  ├ ✓ write-agents-tests  agent   claude-sonnet-4-6  10t ↑28 ↓13k $0.030
  └ ✓ fix-stability       agent   claude-sonnet-4-6  10t ↑13 ↓3.9k $0.012
    ✓ verify              gate    BLOCK 3 type errors in test files  deepseek-v4-flash
    ⊘ report              reduce  skipped · Gate blocked  ↳ fix-stability
```

**लेआउट ही DAG है।** कोई डैशबोर्ड नहीं, grep करने के लिए कोई लॉग नहीं — आप प्रगति पट्टी पढ़ते हैं और पूरी पाइपलाइन समझ जाते हैं:

- **हेडर** — `⊗` = अवरुद्ध (एक गेट ने इसे रोका); `6/7` चरण संसाधित; कुल लागत `$0.095`।
- **स्थिति आइकन** — `✓` पूर्ण · `◐` चल रहा · `✗` विफल · `⊘` छोड़ा गया · `○` लंबित।
- **रेल `┌ ├ └`** — एक ही DAG स्तर में चरण, समवर्ती रूप से चल रहे हैं। चार `write-*`/`fix-stability` कार्य `discover` से फैन आउट होते हैं। एक खाली गटर = एकल-चरण स्तर।
- **`↳`** — एक लंबी, स्तर-छोड़ने वाली निर्भरता। `report` आसन्न `verify` *और* दो स्तर पीछे `fix-stability` पर निर्भर करता है, इसलिए केवल उस स्किप किनारे को एनोटेट किया गया है।
- **गेट** — `verify` ने `VERDICT: BLOCK` उत्सर्जित किया, इसलिए रनटाइम ने `report` छोड़ दिया और रन को `blocked` के रूप में समाप्त किया, कारण इनलाइन दिखाते हुए।
- **विवरण** — प्रति चरण: मॉडल, टोकन गणनाएँ (`↑`इन `↓`आउट), लागत, समय। फैन-आउट चरण उप-कार्य प्रगति भी दिखाते हैं (`3/15 2✗ 8▸`)।

## घोषणात्मक बनें

शॉर्टहैंड आपका प्रवेश मार्ग है। DSL वह जगह है जहाँ `pi-taskflow` अपनी उपयोगिता साबित करता है — डायनामिक फैन-आउट, संरचित रूटिंग, और गुणवत्ता गेट।

### फैन आउट और रिड्यूस

```jsonc
{
  "name": "summarize-files",
  "description": "फ़ाइलें खोजें, प्रत्येक का सारांश दें, एक रिपोर्ट तैयार करें",
  "args": { "dir": { "default": "." } },
  "concurrency": 8,
  "phases": [
    { "id": "discover", "type": "agent", "agent": "scout",
      "task": "{args.dir} के अंतर्गत स्रोत फ़ाइलों को सूचीबद्ध करें (गैर-पुनरावर्ती)।\nकेवल JSON ऐरे आउटपुट करें [{\"file\":\"\"}]। कोई गद्य नहीं।",
      "output": "json" },
    { "id": "summarize", "type": "map",
      "over": "{steps.discover.json}", "as": "item", "agent": "scout",
      "task": "{item.file} पढ़ें और एक-वाक्य का सारांश दें।",
      "dependsOn": ["discover"] },
    { "id": "report", "type": "reduce", "from": ["summarize"], "agent": "writer",
      "task": "एक संक्षिप्त अवलोकन में संयोजित करें:\n{steps.summarize.output}",
      "dependsOn": ["summarize"], "final": true }
  ]
}
```

1. **`discover`** हर फ़ाइल को सूचीबद्ध करता है और एक JSON ऐरे उत्सर्जित करता है।
2. **`summarize`** एक `map` है — यह प्रति फ़ाइल एक उप-एजेंट फैन आउट करता है, 8 समवर्ती तक सीमित, प्रत्येक पथ से `{item.file}` बाध्य होता है।
3. **`report`** एक `reduce` है — यह हर सारांश को एक साफ़ अवलोकन में मर्ज करता है।

मध्यवर्ती सारांश कभी आपके कॉन्टेक्स्ट में प्रवेश नहीं करते। रनटाइम उनका मालिक है; आपको रिपोर्ट मिलती है। **एक बार सहेजें → `/tf:summarize-files dir=src` हमेशा के लिए।**

### रूट करें, गेट करें, पुनर्प्रयास करें, अनुमोदन करें, और व्यय सीमित करें

```jsonc
{
  "name": "triage-and-fix",
  "budget": { "maxUSD": 1.5 },
  "phases": [
    { "id": "triage", "type": "agent", "agent": "analyst", "output": "json",
      "task": "बग को वर्गीकृत करें। केवल आउटपुट करें {\"severity\":\"high\"} या {\"severity\":\"low\"}।" },
    { "id": "deep",  "when": "{steps.triage.json.severity} == high", "dependsOn": ["triage"],
      "agent": "executor-code", "task": "मूल कारण ढूँढें और इसे पैच करें।",
      "retry": { "max": 2, "backoffMs": 500 } },
    { "id": "quick", "when": "{steps.triage.json.severity} == low",  "dependsOn": ["triage"],
      "agent": "executor-fast", "task": "त्वरित सुधार लागू करें।" },
    { "id": "approve", "type": "approval", "join": "any", "dependsOn": ["deep", "quick"],
      "task": "शिप करने से पहले सुधार की समीक्षा करें।" },
    { "id": "ship", "type": "agent", "dependsOn": ["approve"],
      "task": "परिवर्तन के साथ एक PR खोलें।", "final": true }
  ]
}
```

- **`when`** ट्राइएज JSON से `deep` *या* `quick` पर रूट करता है — दूसरी शाखा छोड़ दी जाती है।
- **`join: "any"`** `approve` को उसी क्षण फायर करने देता है जब कोई भी शाखा पूरी होती है (एक OR-जॉइन)।
- **`retry`** बैकऑफ़ के साथ एक अस्थिर पैच को पुनः चलाता है; **`budget`** पूरे रन को रोक देता है यदि वह बहुत महंगा हो जाता है।
- **`approval`** एक मानव के लिए रुकता है (स्वीकृत / अस्वीकृत / संपादित) अंतिम `ship` से पहले।

कोई स्क्रिप्टिंग नहीं। कोई `eval` नहीं। बस डेटा जिसे रनटाइम निष्पादित करता है — LLM-जनित परिभाषाओं को सीधे चलाने के लिए पर्याप्त सुरक्षित।

## चरण प्रकार (Phase types)

| प्रकार | यह क्या करता है | आवश्यक फ़ील्ड |
|------|--------------|-----------------|
| `agent` | एक उप-एजेंट एकल कार्य चलाता है | `task` |
| `parallel` | `branches[]` को समवर्ती रूप से चलाएँ | `branches` ({`task`, `agent?`} की ऐरे) |
| `map` | एक ऐरे पर **फैन आउट** — प्रति आइटम एक उप-एजेंट, `{item}` बाध्य | `over`, `task` |
| `gate` | गुणवत्ता/समीक्षा चरण जो **फ़्लो को रोक सकता है** | `task` |
| `reduce` | `from[]` चरण आउटपुट को एक में एकत्रित करें | `from`, `task` |
| `approval` | **मानव-इन-द-लूप** विराम — स्वीकृत / अस्वीकृत / संपादित | — |
| `flow` | एक **सहेजे गए उप-फ़्लो** को एक चरण के रूप में चलाएँ (संरचना) | `use` |
| `loop` | **किसी कार्य को तब तक दोहराएँ जब तक पूरा न हो** — एक कार्य बॉडी को तब तक पुनः चलाएँ जब तक कोई शर्त, अभिसरण, या सीमा न लगे | `task`, `until` |
| `tournament` | **N प्रकार प्रतिस्पर्धा करते हैं**, एक न्यायाधीश सर्वश्रेष्ठ चुनता है (या एकत्रित करता है) | `task` \| `branches` |

### सामान्य चरण फ़ील्ड

प्रत्येक चरण को एक अद्वितीय `id` और एक `type` की आवश्यकता होती है (डिफ़ॉल्ट `agent` है)। प्रति-प्रकार फ़ील्ड के अतिरिक्त:

| फ़ील्ड | अर्थ |
|---|---|
| `agent` | चलाने के लिए एजेंट (डिफ़ॉल्ट पहले खोजे गए एजेंट का उपयोग करता है) |
| `dependsOn` | चरण ids जिनकी यह चरण प्रतीक्षा करता है — DAG बनाता है |
| `join` | `"all"` (डिफ़ॉल्ट) हर निर्भरता की प्रतीक्षा करता है; `"any"` एक OR-जॉइन है |
| `when` | सशर्त गार्ड — चरण को तब तक छोड़ें जब तक अभिव्यक्ति सत्य न हो |
| `retry` | `{ max, backoffMs?, factor? }` — विफल उप-एजेंट को पुनः प्रयास करें |
| `output` | `"text"` (डिफ़ॉल्ट) या `"json"` (`{steps.ID.json}` को उजागर करता है) |
| `model` / `thinking` / `tools` | उप-एजेंट के लिए प्रति-चरण ओवरराइड |
| `cwd` | उप-एजेंट के लिए कार्यशील निर्देशिका |
| `concurrency` | `map` / `parallel` के लिए फैन-आउट सीमा (फ़्लो डिफ़ॉल्ट को ओवरराइड करती है) |
| `final` | परिणाम-वहन करने वाले चरण को चिह्नित करता है (अन्यथा अंतिम चरण जीतता है) |
| `optional` | यहाँ विफलता रन को **समाप्त नहीं** करती |
| `use` / `with` | (`flow`) सहेजे गए उप-फ़्लो का नाम + इसके तर्क |
| `cache` | `{ scope, ttl?, fingerprint? }` — क्रॉस-रन मेमोइज़ेशन (नीचे देखें) |

फ़्लो-स्तरीय कुंजियाँ: `name`, `description`, `args`, `concurrency` (डिफ़ॉल्ट 8), `agentScope`, और `budget: { maxUSD?, maxTokens? }`।

### नियंत्रण प्रवाह और विश्वसनीयता

- **`when`** — किसी चरण को तब तक छोड़ें जब तक अभिव्यक्ति सत्य न हो। `{refs}`, `== != < > <= >=`, `&& || !`, कोष्ठक, और उद्धृत स्ट्रिंग्स/संख्या समर्थित हैं। मर्ज चरण पर `join: "any"` के साथ जोड़कर वास्तविक if/else रूटिंग करें। पार्स त्रुटियाँ **फ़ेल ओपन** होती हैं।
- **`join: "any"`** — एक OR-जॉइन: चरण जैसे ही *एक* निर्भरता पूरी होती है चलता है (डिफ़ॉल्ट `"all"` सभी की प्रतीक्षा करता है)।
- **`retry`** — `{ "max": 2, "backoffMs": 500, "factor": 2 }` एक विफल उप-एजेंट को निश्चित या घातांकीय बैकऑफ़ के साथ पुनः प्रयास करता है; उपयोग संचित होता है और प्रयास संख्या TUI में `↻N` के रूप में दिखती है। क्षणिक प्रदाता त्रुटियाँ (दर-सीमा / 5xx / टाइमआउट) **स्पष्ट नीति के बिना भी ऑटो-रीट्राई** करती हैं; कठोर त्रुटियाँ नहीं करतीं।
- **`approval`** — एक मानव के लिए रुकें (स्वीकृत / अस्वीकृत / संपादित)। अस्वीकृत फ़्लो को रोकता है; संपादित डाउनस्ट्रीम चरणों के लिए चरण आउटपुट के रूप में टाइप किया गया नोट इंजेक्ट करता है। गैर-इंटरैक्टिव रन ऑटो-अप्रूव करते हैं।
- **`flow`** — `{ "type": "flow", "use": "deep-research", "with": { "topic": "{item}" } }` एक सहेजे गए फ़्लो को एक चरण के रूप में चलाता है (पुनरावृत्ति का पता लगाकर अस्वीकार किया जाता है)।

### लूप-अनटिल-डन (`loop`)

कुछ कार्य स्वाभाविक रूप से पुनरावृत्त होते हैं — ड्राफ़्ट को तब तक परिष्कृत करें जब तक समीक्षक संतुष्ट न हो, तब तक पुनः प्रयास करें और सुधारें जब तक परीक्षण पास न हो जाएँ, किसी उत्तर पर अभिसरण करें। एक `loop` चरण एक कार्य बॉडी को तब तक पुनः चलाता है जब तक कोई रुकने की शर्त लागू न हो:

```jsonc
{
  "id": "refine",
  "type": "loop",
  "task": "इस ड्राफ़्ट में सुधार करें (पुनरावृत्ति {loop.iteration})। पिछला प्रयास:\n{loop.lastOutput}\n\nJSON लौटाएँ {\"draft\":\"…\",\"done\":true|false}।",
  "until": "{steps.refine.json.done} == true",
  "output": "json",
  "maxIterations": 6,
  "convergence": true
}
```

- **बॉडी लोकल** — कार्य `{loop.iteration}` (1-आधारित), `{loop.lastOutput}` (पिछली पुनरावृत्ति का आउटपुट), और `{loop.maxIterations}` पढ़ सकता है; तीनों `until` शर्त के लिए भी उपलब्ध हैं।
- **`until`** — प्रत्येक पुनरावृत्ति के बाद मूल्यांकित, पुनरावृत्ति का आउटपुट `{steps.<thisId>.output}` / `.json` के रूप में उजागर होता है। `when` के समान ऑपरेटर। लूप उसी क्षण रुक जाता है जब यह सत्य हो।
- **हमेशा समाप्त होता है।** चार स्वतंत्र रुकने की स्थितियाँ: `until` सत्य, **अभिसरण** (एक निश्चित बिंदु — आउटपुट पिछली पुनरावृत्ति के समान), **`maxIterations`** (अधिकतम 100 पर हार्ड-कैप्ड), या एक **विफल पुनरावृत्ति** (चरण आंशिक आउटपुट के साथ विफल होता है)। एक दोषपूर्ण `until` हमेशा के लिए घूमने के बजाय लूप को **रोकता** है (फ़ेल-सेफ) और चरण पर एक चेतावनी दिखाता है।
- TUI `↻N` को रुकने के कारण (`done` / `converged` / `max` / `failed`) के साथ दिखाता है; उपयोग पुनरावृत्तियों में संचित होता है। `gate`/`approval` की तरह, `loop` **`cross-run` कैश से बाहर रखा गया** है (प्रत्येक रन को ताज़ा पुनरावृत्त करना चाहिए)।

### टूर्नामेंट (`tournament`)

खुले-अंत वाले कार्यों के लिए, सबसे अच्छा परिणाम अक्सर कई उम्मीदवार उत्पन्न करने और सबसे मजबूत चुनने से आता है — एक न्यायाधीश के साथ बेस्ट-ऑफ़-N, एक घोषणात्मक चरण में:

```jsonc
{
  "id": "headline",
  "type": "tournament",
  "task": "इस लॉन्च पोस्ट के लिए एक आकर्षक हेडलाइन लिखें।",
  "variants": 4,
  "judge": "सबसे मजबूत हुक और स्पष्टतम वादे वाली हेडलाइन चुनें।",
  "judgeAgent": "reviewer",
  "mode": "best"
}
```

- **प्रतियोगी** — या तो एक `task` की `variants: N` प्रतियाँ (विविधता मॉडल नॉनडेटरमिनिज्म से आती है), या अलग `branches: [{task, agent?}, …]` जब आप *विभिन्न दृष्टिकोणों* को एक-दूसरे के खिलाफ खड़ा करना चाहते हैं।
- **न्यायाधीश** — फैन-आउट के बाद, एक न्यायाधीश एजेंट हर वेरिएंट (क्रमांकित) और आपका `judge` मानदंड देखता है और `WINNER: <n>` पंक्ति या `{"winner": n}` के माध्यम से विजेता चुनता है। एक अपठनीय निर्णय वेरिएंट 1 पर **फ़ेल ओपन** करता है; एक विफल न्यायाधीश भी वापस गिर जाता है — कार्य कभी खोता नहीं है।
- **`mode`** — `best` विजेता वेरिएंट को **शब्दशः** लौटाता है; `aggregate` न्यायाधीश का **संश्लेषित** उत्तर लौटाता है जो सबसे मजबूत भागों को जोड़ता है।
- **शॉर्ट-सर्किट:** यदि केवल एक प्रतियोगी बचता है, तो वह बिना न्यायाधीश कॉल के जीतता है; यदि सभी विफल होते हैं, तो चरण विफल होता है। TUI `⚑ N→#k` दिखाता है; उपयोग वेरिएंट + न्यायाधीश संचित करता है। `gate` की तरह, यह **`cross-run` कैश से बाहर रखा गया** है।
- **`budget`** — एक रन-व्यापी `{maxUSD, maxTokens}` सीमा; एक बार पार होने पर, लंबित चरण छोड़ दिए जाते हैं और इन-फ़्लाइट फैन-आउट नए स्पॉन करना बंद कर देता है, रन को `blocked` के रूप में समाप्त करता है।
- **आइडल वॉचडॉग** — एक उप-एजेंट जो 5 मिनट तक चुप रहता है उसे अटका हुआ मानकर मार दिया जाता है (SIGTERM → SIGKILL), ताकि एक लटका हुआ चाइल्ड कभी भी पूरे फ़्लो को फ़्रीज़ न कर सके।

### क्रॉस-रन मेमोइज़ेशन (`cache`)

प्रत्येक चरण पहले से ही कंटेंट-एड्रेस्ड है: एकल रन के **पुनर्प्रारंभ** के भीतर, एक चरण जिसके हल किए गए इनपुट अपरिवर्तित हैं, छोड़ दिया जाता है। `cache` उस पुन: उपयोग को **स्वतंत्र रनों में** विस्तारित करता है — यदि किसी पिछले रन ने समान इनपुट हैश वाले चरण की गणना की है, तो इसका परिणाम **$0.00** के लिए पुन: उपयोग किया जाता है।

```jsonc
{
  "id": "analyze-auth",
  "task": "बताएँ कि auth मॉड्यूल कैसे काम करता है।",
  "context": ["src/auth/**/*.ts"],
  "cache": {
    "scope": "cross-run",
    "ttl": "6h",
    "fingerprint": ["git:HEAD", "glob:src/auth/**/*.ts"]
  }
}
```

- **`scope`** — `"run-only"` (डिफ़ॉल्ट) बिल्कुल ऐतिहासिक व्यवहार है (केवल रन-के-भीतर पुनर्प्रारंभ)। `"cross-run"` चरण को स्थायी भंडार में चुनता है। `"off"` पुन: उपयोग को पूरी तरह से अक्षम करता है (एक रन के भीतर भी), डीबगिंग के लिए।
- **ताज़गी ही पूरा खेल है।** कैश कुंजी में पहले से ही प्रॉम्प्ट, `over` आइटम, और कोई भी `context` फ़ाइल शामिल है। `fingerprint` *अंतर्निहित* इनपुट को कुंजी में शामिल करता है ताकि "दुनिया बदल गई" कैश मिस बन जाए: `git:HEAD`, `glob:<pat>` (आकार+mtime), `glob!:<pat>` (सामग्री हैश), `file:<path>`, `env:<NAME>`। `ttl` (`30m`/`6h`/`7d`) एक समय समर्थन है।
- **ईमानदार सीमा:** एक उप-एजेंट जो एक फ़ाइल पढ़ता है जिसे उसने `context`/`fingerprint` में घोषित नहीं किया, फिर भी एक पुराना `cross-run` हिट दे सकता है। यही कारण है कि डिफ़ॉल्ट `run-only` है और क्यों `gate`/`approval` चरणों को `cross-run` से **प्रतिबंधित** किया गया है (उन्हें प्रत्येक रन में ताज़ा परिणाम देना चाहिए)। केवल उन चरणों के लिए ऑप्ट इन करें जिनका आउटपुट घोषित इनपुट का फ़ंक्शन है।
- कैश `.pi/taskflows/cache/` (gitignored) में रहता है। `action: "cache-clear"` से साफ़ करें। पूर्ण तर्क: [`docs/rfc-cross-run-memoization.md`](./docs/rfc-cross-run-memoization.md)।

### गेट चरण (गुणवत्ता नियंत्रण)

एक `gate` एक एजेंट को ऊपरी प्रवाह आउटपुट की समीक्षा करने के लिए चलाता है और **शेष वर्कफ़्लो को रोक सकता है।** गेट कार्य को रनटाइम द्वारा पढ़े जाने योग्य निर्णय माँगकर समाप्त करें:

- एक अंतिम पंक्ति `VERDICT: PASS` या `VERDICT: BLOCK` (`OK`, `FAIL`, `STOP`, `REJECT`, `HALT` भी स्वीकार करता है — अंतिम घटना जीतती है), या
- JSON जैसे `{"continue": false, "reason": "missing auth checks"}` / `{"verdict": "block", "reason": "..."}`।

**BLOCK** पर, डाउनस्ट्रीम चरण छोड़ दिए जाते हैं और रन `blocked` के रूप में समाप्त होता है जिसमें कारण दिखाया जाता है। **अस्पष्ट आउटपुट फ़ेल ओपन** (PASS माना जाता है) — एक गेट कभी भी आपके फ़्लो को दुर्घटनावश नहीं रोकता।

```
नीचे दिए गए ऑडिट की समीक्षा करें। यदि कोई एंडपॉइंट auth से छूट गया है, तो
"VERDICT: BLOCK" और एक-पंक्ति कारण के साथ समाप्त करें; अन्यथा "VERDICT: PASS" के साथ समाप्त करें।

{steps.audit.output}
```

## इंटरपोलेशन और अभिव्यक्तियाँ

| प्लेसहोल्डर | हल करता है |
|---|---|
| `{args.X}` | आह्वान तर्क |
| `{steps.ID.output}` | पिछले चरण का टेक्स्ट आउटपुट |
| `{steps.ID.json}` | पिछले आउटपुट को JSON के रूप में पार्स किया गया (या `{steps.ID.json.field}`) |
| `{item}` / `{item.field}` | `map` चरण के अंदर वर्तमान आइटम |
| `{previous.output}` | तुरंत ऊपरी-प्रवाह चरण आउटपुट |

शर्त व्याकरण (`when` के लिए): `== != < > <= >=`, `&& || !`, कोष्ठक, उद्धृत स्ट्रिंग्स/संख्या, और कोई भी `{...}` संदर्भ — जैसे `"when": "{steps.triage.json.route} == deep && {args.force} != true"`।

> `{steps.X}` का संदर्भ जो `dependsOn` में घोषित नहीं है, एक **कठोर मान्यता त्रुटि** है — रनटाइम एक भी एजेंट चलने से पहले सबसे आम पाइपलाइन बग को पकड़ लेता है।

## कमांड्स

सहेजे गए फ़्लो CLI शॉर्टकट बन जाते हैं। सभी कमांड Pi सत्र में चलते हैं:

| कमांड | यह क्या करता है |
|---|---|
| `/tf list` | सभी सहेजे गए फ़्लो सूचीबद्ध करें |
| `/tf run <name> [args]` | एक सहेजा गया फ़्लो चलाएँ (जैसे `/tf run summarize-files dir=src`) |
| `/tf show <name>` | एक फ़्लो की परिभाषा प्रिंट करें |
| `/tf runs` | हाल के रन इतिहास ब्राउज़ करें (इंटरैक्टिव TUI) |
| `/tf resume <runId>` | एक रुके/विफल रन को जारी रखें — कैश्ड चरण अपने-आप छोड़े जाते हैं |
| `/tf init` | **मॉडल भूमिकाओं को इंटरैक्टिव रूप से मैप करें** आपके सक्षम मॉडलों से (`~/.pi/agent/settings.json` लिखता है) |
| `/tf:<name> [args]` | शॉर्टकट — फ़्लो को एक टैप में चलाता है |

टूल क्रियाएँ (मॉडल द्वारा उपयोग): `run` (इनलाइन `define` या सहेजा गया `name`), `save`, `resume`, `list`, `init`।

## सत्रों में पुनर्प्रारंभ

## भंडारण

```
.pi/taskflows/<name>.json          # परियोजना-स्कोप्ड परिभाषाएँ (साझा करने के लिए कमिट करें)
~/.pi/agent/taskflows/<name>.json  # उपयोगकर्ता-स्कोप्ड परिभाषाएँ
.pi/taskflows/runs/<runId>.json    # पुनर्प्रारंभ के लिए रन स्थिति (इसे gitignore करें)
```

> `.pi/taskflows/` कमिट करें और आपकी पूरी टीम पाइपलाइन साझा करती है — कोई कॉन्फ़िग सिंक नहीं, कोई ऑनबोर्डिंग दस्तावेज़ नहीं। रन स्थिति परमाणु रूप से लिखी जाती है और एक शून्य-निर्भरता फ़ाइल लॉक द्वारा संरक्षित होती है, ताकि समवर्ती रन कभी इंडेक्स को दूषित न करें।

एजेंट खोज क्षेत्र (फ़्लो परिभाषा में `agentScope` के माध्यम से):

| मान | एजेंटों को खोजता है |
|---|---|
| `"user"` (डिफ़ॉल्ट) | `~/.pi/agent/agents/*.md` |
| `"project"` | `.pi/agents/*.md` (ट्री ऊपर चलता है) |
| `"both"` | उपयोगकर्ता + परियोजना; नाम टकराव पर परियोजना जीतती है |

## एजेंट

Taskflow में **18 बिल्ट-इन एजेंट** हैं — प्रत्येक एक `.md` फ़ाइल जिसमें ट्यून किया गया सिस्टम प्रॉम्प्ट, थिंकिंग स्तर, और टूल सेट है। आप किसी भी चरण या शॉर्टहैंड में उन्हें `name` द्वारा संदर्भित कर सकते हैं, इंस्टॉल के तुरंत बाद। कोई सेटअप आवश्यक नहीं।

| `plan-arbiter` | योजना गुणवत्ता गेट (जटिल कार्य) | उच्च | `{{arbiter}}` |
| `final-arbiter` | जब समीक्षक असहमत हों तो टाईब्रेकर | अत्यधिक उच्च | `{{arbiter}}` |
| `test-engineer` | परीक्षणों का डिज़ाइन और कार्यान्वयन | उच्च | `{{fast}}` |
| `doc-writer` | दस्तावेज़ीकरण लेखन | बंद | `{{fast}}` |
| `recover` | संघनन के बाद सत्र पुनर्प्राप्ति | निम्न | `{{fast}}` |
| `verifier` | परीक्षण चलाएँ, परिणाम मान्य करें | बंद | `{{fast}}` |
| `visual-explorer` | Figma डिज़ाइन मेटाडेटा विश्लेषण | उच्च | `{{vision}}` |

एजेंट स्तरित हैं: **बिल्ट-इन → उपयोगकर्ता (`~/.pi/agent/agents/`) → परियोजना (`.pi/agents/`)**। समान `name` वाला उपयोगकर्ता या परियोजना एजेंट बिल्ट-इन को ओवरराइड करता है — ताकि आप पैकेज को छुए बिना किसी भी एजेंट को अनुकूलित कर सकें।

### मॉडल भूमिकाएँ (Model roles)

प्रत्येक बिल्ट-इन एजेंट का `model` फ़ील्ड एक **भूमिका प्लेसहोल्डर** (जैसे `{{fast}}`) का उपयोग करता है न कि हार्डकोडेड प्रदाता स्ट्रिंग का। यह *आशय* को *कार्यान्वयन* से अलग करता है — आप भूमिकाओं को एक बार मॉडलों से मैप करते हैं, और हर एजेंट अनुकूलित हो जाता है।

| भूमिका | आशय | विशिष्ट मॉडल |
|---|---|---|
| `{{fast}}` | सस्ता और तेज़ — उच्च-मात्रा, कम-दाँव | DeepSeek V4 Flash |
| `{{strong}}` | संतुलित — योजना, समीक्षा, मध्यम जटिलता | MiMo v2.5 Pro |
| `{{thinker}}` | गहन विश्लेषण — आवश्यकताएँ, आलोचना | DeepSeek V4 Pro |
| `{{arbiter}}` | अंतिम निर्णय — टाईब्रेक, योजना गुणवत्ता गेट | Qwen 3.7 Max |
| `{{vision}}` | मल्टीमॉडल — UI कार्य, डिज़ाइन पढ़ना | MiniMax M3 |
| `{{reasoner}}` | सतर्क तर्क — सुरक्षा, जोखिम | GLM 5.1 |

कॉन्फ़िगरेशन के बिना, एजेंट Pi के डिफ़ॉल्ट मॉडल पर वापस गिर जाते हैं। भूमिकाओं को वास्तविक मॉडलों से मैप करने के लिए, इंटरैक्टिव सेटअप चलाएँ:

```bash
/tf init
```

`/tf init` एक **क्रिया मेनू** से शुरू होता है। पहली बार उपयोगकर्ताओं को 2-विकल्प शॉर्टकट मिलता है ("अनुशंसित डिफ़ॉल्ट का उपयोग करें" / "प्रत्येक भूमिका कॉन्फ़िगर करें")। वापस लौटने वाले उपयोगकर्ता पूरा 5-विकल्प मेनू देखते हैं:

```
? मॉडल भूमिकाओं के साथ आप क्या करना चाहते हैं?
  ❯ अनुशंसित डिफ़ॉल्ट का उपयोग करें
    प्रत्येक भूमिका कॉन्फ़िगर करें
    एक भूमिका संपादित करें
    वर्तमान भूमिकाएँ दिखाएँ
    रद्द करें
```

पिकर मॉडल **प्रदर्शन नाम** क्षमता ध्वज और वर्तमान/अनुशंसित मार्कर के साथ दिखाता है:

```
? 'vision' के लिए मॉडल — मल्टीमॉडल (executor-ui, visual-explorer)
  वर्तमान: openrouter/anthropic/claude-sonnet-4-6
  अनुशंसित: minimax/MiniMax-M3
  ───────────────
  ❯ MiniMax M3 (minimax/MiniMax-M3) · image ✓ · reasoning ✓ · (अनुशंसित)
    Claude Sonnet 4.6 (openrouter/anthropic/...) · image ✓ · reasoning ✓ · (वर्तमान)
    GPT-5 (openrouter/openai/gpt-5) · image ✓
    DeepSeek V4 Flash (openrouter/deepseek/v4-flash)
    ───────────────
    कस्टम (अपना स्वयं का टाइप करें)
    वर्तमान रखें
    क्रिया मेनू पर वापस जाएँ
```

सहेजने से पहले, एक **पूर्वावलोकन स्क्रीन** आपके परिवर्तनों का अंतर दिखाती है:

```
? परिवर्तनों की समीक्षा करें:
  fast       openrouter/deepseek/deepseek-v4-flash   (अपरिवर्तित)
  strong     openrouter/xiaomi/mimo-v2.5-pro         (अपरिवर्तित)
  thinker    openrouter/qwen/qwen3.7-max             (बदला ← था: openrouter/deepseek/v4-pro)
  arbiter    openrouter/qwen/qwen3.7-max             (अपरिवर्तित)
  vision     minimax/MiniMax-M3                      (अपरिवर्तित)
  reasoner   z-ai/glm-5.1                            (अपरिवर्तित)
  ───────────────
  ❯ ये परिवर्तन सहेजें
    एक भूमिका संपादित करें
    रद्द करें
```

आपके चयन `~/.pi/agent/settings.json` में लिखे जाते हैं:

```json
{
  "modelRoles": {
    "fast":     "openrouter/deepseek/deepseek-v4-flash",
    "strong":   "openrouter/xiaomi/mimo-v2.5-pro",
    "thinker":  "openrouter/deepseek/deepseek-v4-pro",
    "arbiter":  "openrouter/qwen/qwen3.7-max",
    "vision":   "minimax/MiniMax-M3",
    "reasoner": "z-ai/glm-5.1"
  }
}
```

मानों को किसी भी समय मैन्युअल रूप से संपादित करें, या बस `/tf init` पुनः चलाएँ। आप उसी फ़ाइल में `subagents.agentOverrides` के माध्यम से व्यक्तिगत एजेंटों को भी ओवरराइड कर सकते हैं:

```json
{
  "modelRoles": { ... },
  "subagents": {
    "agentOverrides": {
      "executor": { "model": "anthropic/claude-sonnet-4-20250514" },
      "reviewer": { "thinking": "xhigh" }
    }
  }
}
```

### टूल पथ (`action="init"`)

मॉडल `taskflow` टूल के माध्यम से भूमिकाएँ भी कॉन्फ़िगर कर सकता है:

| मोड | व्यवहार |
|---|---|
| `mode: "show"` (डिफ़ॉल्ट) | वर्तमान `modelRoles` की केवल-पढ़ने वाली रिपोर्ट। कभी ओवरराइट नहीं करता। |
| `mode: "apply-defaults"` + `force: true` | पुरानी कुंजियों को संरक्षित करते हुए `settings.json` में `RECOMMENDED_DEFAULTS` लिखता है। |
| `mode: "interactive"` | पूर्ण क्रिया मेनू + पिकर फ़्लो लॉन्च करता है (UI सत्र की आवश्यकता है)। |

> **v0.0.13 बहिष्करण नोट:** यदि `mode` छोड़ा गया है, तो टूल v0.0.12 व्यवहार पर वापस गिर जाता है जब `modelRoles` खाली है (ऑटो-राइट डिफ़ॉल्ट) एक `console.warn` बहिष्करण सूचना के साथ। यदि `modelRoles` पहले से मौजूद है, तो यह `mode: "show"` के रूप में व्यवहार करता है। यह पुल v0.0.14 में हटा दिया जाएगा।

### कस्टम एजेंट

अपना स्वयं का जोड़ने के लिए `~/.pi/agent/agents/` (उपयोगकर्ता-स्तर) या `.pi/agents/` (परियोजना-स्तर, इसे कमिट करें) में एक `.md` फ़ाइल डालें:

```markdown
---
name: my-linter

description: ESLint चलाएँ और उल्लंघन रिपोर्ट करें

tools: read, bash

model: "{{fast}}"

thinking: off
---

आप एक लिंटिंग एजेंट हैं। प्रदान की गई फ़ाइलों पर `npx eslint --format json` चलाएँ।
उल्लंघनों को फ़ाइल के अनुसार समूहित करके रिपोर्ट करें। कोई सुधार नहीं।
```

फिर किसी भी चरण में संदर्भित करें: `{ "agent": "my-linter", "task": "src/ को लिंट करें" }`।

## 🍽️ हम अपना स्वयं का डॉग फ़ूड खाते हैं

`pi-taskflow` की हर सुविधा **`pi-taskflow` के माध्यम से शिप** होती है।

हमारा `self-improve` फ़्लो एक 10-चरण DAG है — यह कोडबेस का ऑडिट करता है, दोषों को पैच करता है, शुद्धता सत्यापित करता है, गुणवत्ता पर गेट करता है, और रिपोर्ट सतह पर लाता है — सब घोषणात्मक रूप से। इसे `/tf:self-improve` के रूप में सहेजा गया है और हर रिलीज़ से पहले चलाया जाता है। Pi इकोसिस्टम में कोई अन्य एजेंट ऑर्केस्ट्रेटर स्वयं को स्वयं से नहीं बनाता।

| अभियान | पैमाना | चरण | परिणाम |
|----------|-------|--------|---------|
| [v0.0.8 dogfood](./docs/dogfooding-v0.0.8-report.md) | पूर्ण कोडबेस ऑडिट → ट्राइएज → सुधार → सत्यापन | 10 चरण, 234 परीक्षण | 13 सुधार, सभी पास |
| [v0.0.6 self-audit](./docs/self-audit-report.md) | इन्वेंटरी → मैप ऑडिट → गेट → अनुमोदन → मैप सुधार → रिड्यूस | 9 चरण | 11 महत्वपूर्ण दोष ठीक किए गए |
| [क्रॉस-रन कैश डॉगफ़ूड](./docs/rfc-cross-run-memoization.md) | वास्तविक रनटाइम + ऑन-डिस्क स्टोर | समर्पित परीक्षण हार्नेस | प्रतिकूल फ़िंगरप्रिंट के तहत कैश शुद्धता |

> **मेटा:** हमने `pi-taskflow` के `map` फैन-आउट, `gate` निर्णय, `approval` मानव-इन-द-लूप, `tournament` बेस्ट-ऑफ़-N, `loop` अनटिल-डन, और `cross-run` कैश का उपयोग किया — `pi-taskflow` बनाने के लिए।

## स्थिति और सीमाएँ

**v0.0.13** — लूप-अनटिल-डन (`loop` चरण: एक शर्त, अभिसरण, या सीमा तक पुनरावृत्ति), टूर्नामेंट (न्यायाधीश के साथ बेस्ट-ऑफ़-N), क्रॉस-रन मेमोइज़ेशन (git/फ़ाइल/glob/env फ़िंगरप्रिंट और TTL के साथ सामग्री-संबोधित कैश), इंटरैक्टिव `/tf init` भूमिका-जागरूक मॉडल पिकर + अंतर पूर्वावलोकन + परमाणु मर्ज-राइट के साथ, 18 बिल्ट-इन एजेंट 6 मॉडल भूमिकाओं के साथ। DSL + DAG रनटाइम (`agent`/`parallel`/`map`/`gate`/`reduce`) के ऊपर पूर्ण नियंत्रण-प्रवाह और विश्वसनीयता स्तर (`when` गार्ड, `join: any`, `retry`/बैकऑफ़, `approval`, `flow` संरचना, `budget` सीमाएँ, आइडल वॉचडॉग)। इनलाइन + सहेजे गए फ़्लो, क्रॉस-सेशन पुनर्प्रारंभ, लाइव प्रगति, और पृथक कॉन्टेक्स्ट। एक रन एक स्ट्रीमिंग टूल कॉल के रूप में निष्पादित होता है।

ज्ञात सीमाएँ (ट्रैक की गईं, बाध्य — फ़्लो के बीच में कोई आश्चर्य नहीं):

- **कोई अलग पृष्ठभूमि निष्पादन नहीं।** एक रन को Pi सत्र खुला रखने की आवश्यकता है। वास्तविक पृष्ठभूमि निष्पादन (और इसके ऊपर इवेंट/क्रॉन ट्रिगर) रोडमैप पर है।
- **कोई `output: "file"` नहीं।** आउटपुट केवल टेक्स्ट/JSON हैं — एक एजेंट के `write` टूल कॉल के माध्यम से फ़ाइलें लिखें।
- **`map` को JSON ऐरे की आवश्यकता है।** `over` फ़ील्ड को एक `{steps.ID.json}` ऐरे में हल होना चाहिए। एक टेक्स्ट सूची को पहले एकल-एजेंट `output: "json"` चरण में लपेटें।
- **DAG चक्रीय नहीं होना चाहिए।** चक्र मान्यता पर अस्वीकार किए जाते हैं।

## विकास

```bash
npm install

npm test            # यूनिट परीक्षण — कोई नेटवर्क नहीं, कोई प्रक्रिया स्पॉनिंग नहीं
```
