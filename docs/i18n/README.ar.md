---

**الوكلاء الفرعيون (subagents) هم إطلاق وانسَ. تدفقات العمل (taskflows) تُطلق، وتُوزع، وتتوقف، وتمر عبر البوابات، وتستأنف، وتحفظ نفسها كأمر.**

أنت تعرف بالفعل أداة `task` / `tasks` / `chain` المدمجة للوكيل الفرعي. `pi-taskflow` تستخدم نفس الاختصار — لذا فإن تفويضاتك الحالية تصبح فورًا **مُتتبّعة، وقابلة للاستئناف، وقابلة للحفظ كأمر بكلمة واحدة `/tf:<name>`**. عندما تتجاوز الاختصار، تمنحك لغة DSL الكاملة DAG حقيقي: توزيع ديناميكي على عشرات العناصر، توجيه شرطي، بوابات جودة، موافقات بشرية، إعادة محاولة، وسقف إنفاق صارم.

وطوال الوقت، **تصل المرحلة النهائية فقط إلى محادثتك.** كل سجل وسيط يبقى في بيئة التشغيل (runtime)، ولا يدخل أبدًا نافذة السياق الخاصة بك.

## لماذا هذا موجود

ها هو الجدار الذي تصطدم به مع الوكلاء الفرعيين الخام: تصف خطة متعددة الخطوات نثرًا، يعيد النموذج اشتقاقها في كل مرة، تغمر السجلات الوسيطة سياقك، وعندما يفشل استدعاء نموذج واحد تبدأ من الصفر. لا إعادة استخدام، لا استرداد، لا هيكل.

`pi-taskflow` تنقل الخطة **خارج المطالبة وإلى تعريف تصريحي.** بيئة التشغيل تمتلك DAG، والحلقات، وإعادة المحاولة، والحالة الوسيطة. تعلن عن pipeline مرة واحدة وتشغلها مئة مرة — بالاسم.

> عندما تحتاج مهمة إلى اثنتي عشرة خطوة مع توزيع متفرع وبوابة مراجعة، فأنت تريد تنظيمًا — لا مطالبة محظوظة.

| | وكيل فرعي (مدمج) | **pi-taskflow** |
|---|---|---|
| **من يقود** | النموذج، خطوة بخطوة | بيئة التشغيل، من تعريف |
| **الهيكل** | سلسلة / متوازٍ مسطح | **DAG مع تزامن متعدد الطبقات + توجيه** |
| **النتائج الوسيطة** | في نافذة السياق الخاصة بك | **في بيئة التشغيل — وليس سياقك** |
| **النطاق** | حفنة من المهام | **توزيع `map` ديناميكي على عشرات العناصر** |
| **قابل لإعادة الاستخدام** | يُعاد وصفه كل مرة | **محفوظ كـ `/tf:<name>`** |
| **قابل للاستئناف** | ✗ | **✓ عبر الجلسات — المراحل المخبأة تُتخطى تلقائيًا** |
| **بوابات الجودة** | ✗ | **مراحل `gate` تتوقف عند `VERDICT: BLOCK`** |
| **التوجيه الشرطي** | ✗ | **حراس `when` + وصل `join: any` من نوع OR** |
| **تحمل الأخطاء** | ✗ | **إعادة محاولة `retry` لكل مرحلة + إعادة تلقائية للأخطاء العابرة** |
| **إدراج الإنسان في الحلقة** | ✗ | **مراحل `approval` (موافقة / رفض / تحرير)** |
| **التحكم في التكلفة** | ✗ | **ميزانية `budget` على مستوى التشغيل (سقف USD / token)** |
| **التركيب** | ✗ | **مراحل `flow` تشغّل تدفقات فرعية محفوظة** |
| **التقدم المباشر** | مبهم أثناء التشغيل | **عرض DAG مباشر مع التوقيت والتكلفة** |
| **سهولة الاستخدام** | JSON مضمن في كل مرة | **اختصار (`task`/`tasks`/`chain`) أو DSL** |

هذا لا يستبدل أداة الوكيل الفرعي. إنه يمنح وكلاءك الفرعيين DAG وذاكرة واسمًا.

**كيف تختار:**

- **`@pi-agents/orchid`** هو المنظّم الأكثر اكتمالاً في النظام البيئي (DAG + worktrees + حلقة Ralph + صندوق بريد الوكيل) — لكن DSL الخاص به هو pipeline *ثابت* من 9 مراحل، يحمل تبعيات تشغيل + jiti، وهو في مرحلة beta. اختر `pi-taskflow` عندما تريد **تعريف الرسم البياني الخاص بك** (بدلاً من تبني رسم بياني رأي) مع **صفر تبعيات** وتثبيت بأمر واحد.
- **`pi-crew` / `ultimate-pi`** أثقل — عزل worktree، فرق غير متزامنة متينة، حوكمة متعددة المستويات. إذا كنت تريد خفيف الوزن، تصريحي، وبدون تبعيات — هذا هو هذا المشروع.
- **`@zhushanwen/pi-workflow`** هو الأقرب روحًا وأيضًا بدون تبعيات، لكنك تؤلف تدفقات العمل كـ **نصوص JavaScript**. DSL JSON التصريحي لـ `pi-taskflow` أكثر أمانًا وقابلية للتدقيق، واستئنافه بمستوى مرحلة عبر hashing المدخلات أدق من إلغاء التكرار عبر ذاكرة التخزين المؤقت للاستدعاء.
- **`@fiale-plus/pi-rogue-orchestration`** لديه حلقة حقيقية **كرّر حتى الإنجاز** (ميزة لا تمتلكها `pi-taskflow` بعد). إذا كانت مهمتك "استمر حتى يتحقق الهدف"، فهي تستحق النظر؛ `pi-taskflow` مخصصة لـ *pipeline منظمة متفرعة* بدلاً من ذلك.
- **`pi-subagents` / `@gotgenes/pi-subagents`** هي الخيارات الناضجة للتفويض المخصص "استخدم المراجع على هذا الفرق" والمهام الخلفية. `pi-taskflow` مخصصة عندما تحتاج هذه التفويضات أن تصبح *pipeline قابلة للتكرار والاستئناف*.
- **`pi-pipeline` / `pi-agent-flow`** تُصدر تدفقات *رأيية ثابتة*. `pi-taskflow` تُصدر *لوحة فارغة*: أنت (أو النموذج) تُعلن الرسم البياني الذي يناسب المهمة.

> الجملة الصادقة الواحدة: **`pi-taskflow` هي إضافة Pi الوحيدة التي تمنحك pipeline تصريحي وقابل للاستئناف وشكل DAG للوكيل الفرعي تحفظه كأمر بكلمة واحدة — مع صفر تبعيات تشغيل وعزل سياقي حسب التصميم.** الفجوات المعروفة التي تُسد بعد ذلك: حلقة حتى الإنجاز، عزل worktree، وتنفيذ خلفية غير محظورة (انظر [`STRATEGY.md`](./STRATEGY.md)).

## بداية 30 ثانية

**1. التثبيت** — أمر واحد:

```bash
pi install npm:pi-taskflow
```

> **اختياري:** شغّل `/tf init` مرة واحدة لتعيين أدوار النماذج للوكلاء الـ 18 المدمجين
> (`fast`، `strong`، `thinker`، …) لنماذجك الخاصة — منتقي تفاعلي.
> تخطّه وستستخدم الوكلاء النموذج الافتراضي لـ Pi. انظر [أدوار النموذج](#model-roles).

**2. التشغيل** — فقط اطلب من النموذج في جلسة Pi:

> *شغّل سلسلة: أولاً استكشف تدفق المصادقة، ثم لخّص النتائج.*

النموذج يستدعي أداة `taskflow` تلقائيًا. تحصل على تقدم مباشر، وتوقيت لكل خطوة، وتكلفة رمزية، وسجل تشغيل محفوظ — **بنفس جهد الأداة المدمجة، ولكن الآن متتبّع وقابل للاستئناف.**

**3. الحفظ** — قل *"احفظها"* وسيكون لديك `/tf:<name>` للأبد.

هذا كل شيء. يمكنك تشغيل أول تدفق عمل لك قبل أن يبرد قهوتك — دون كتابة تعريف مرحلة واحدة.

### الاختصار (بنفس شكل الأداة المدمجة)

```jsonc
// مفردة — وكيل واحد، مهمة واحدة
{ "task": "لخّص بنية src/", "agent": "explorer" }

// متوازٍ — أطلق عدة دفعة واحدة، المخرجات تندمج
{ "tasks": [
  { "task": "دقّق المصادقة في src/api",             "agent": "analyst" },
  { "task": "دقّق التحقق من الإدخال في src/api", "agent": "analyst" }
] }

// سلسلة — متتابعة؛ كل خطوة ترى مخرجات السابقة
{ "chain": [
  { "task": "اسرد واجهة API العامة لـ src/lib", "agent": "scout" },
  { "task": "اكتب وثائق لـ:\n{previous.output}", "agent": "writer" }
] }
```

`agent` اختياري (الافتراضي هو أول وكيل مكتشف). أضف `name` لتسمية التشغيل وفتح إمكانية حفظه كأمر.

## شاهده أثناء التشغيل

هذه ليست محاكاة. **هذا هو stdout من تشغيل حقيقي** — تدفق `self-improve` الذي يكتب ويتحقق من مجموعات الاختبار الخاصة به، ويُلتقط أثناء الطيران ببوابة جودة:

```
⊗ taskflow self-improve  6/7 · blocked · $0.095
    ✓ discover            agent   deepseek-v4-flash  10t ↑38k ↓6.7k $0.011
  ┌ ✓ write-runner-tests  agent   claude-sonnet-4-6  10t ↑13 ↓6.6k $0.020
  ├ ✓ write-store-tests   agent   claude-sonnet-4-6  10t ↑11 ↓10k $0.018
  ├ ✓ write-agents-tests  agent   claude-sonnet-4-6  10t ↑28 ↓13k $0.030
  └ ✓ fix-stability       agent   claude-sonnet-4-6  10t ↑13 ↓3.9k $0.012
    ✓ verify              gate    BLOCK 3 أخطاء نوع في ملفات الاختبار  deepseek-v4-flash
    ⊘ report              reduce  تم التخطي · البوابة منعت  ↳ fix-stability
```

**التخطيط *هو* DAG.** لا لوحة تحكم، لا سجلات للبحث — تقرأ شريط التقدم وتفهم الـ pipeline بأكمله:

- **الرأس** — `⊗` = محظور (بوابة أوقفته)؛ تمت معالجة `6/7` مراحل؛ التكلفة الإجمالية `$0.095`.
- **أيقونات الحالة** — `✓` تم · `◐` قيد التشغيل · `✗` فشل · `⊘` تم التخطي · `○` معلق.
- **السكة `┌ ├ └`** — مراحل في نفس طبقة DAG، تعمل بشكل متزامن. المهام الأربع `write-*`/`fix-stability` تتفرع من `discover`. حوض فارغ = طبقة مرحلة واحدة.
- **`↳`** — تبعية طويلة تتخطى الطبقات. `report` يعتمد على `verify` المجاور *وعلى* `fix-stability` بعد طبقتين، لذلك يتم توضيح حافة التخطي فقط.
- **البوابة** — أصدرت `verify` `VERDICT: BLOCK`، لذا تخطت بيئة التشغيل `report` وأنهت التشغيل كـ `blocked`، مع عرض السبب في السطر.
- **التفاصيل** — لكل مرحلة: النموذج، عدد الرموز (`↑` داخل `↓` خارج)، التكلفة، التوقيت. مراحل التوزيع تُظهر أيضًا تقدم المهام الفرعية (`3/15 2✗ 8▸`).

## انتقل إلى التصريحية

الاختصار هو المدخل. DSL هو حيث تثبت `pi-taskflow` جدارتها — توزيع ديناميكي، توجيه منظم، وبوابات جودة.

### توزيع وتجميع

```jsonc
{
  "name": "summarize-files",
  "description": "اكتشف الملفات، لخّص كل ملف، أنتج تقريرًا واحدًا",
  "args": { "dir": { "default": "." } },
  "concurrency": 8,
  "phases": [
    { "id": "discover", "type": "agent", "agent": "scout",
      "task": "اسرد ملفات المصدر تحت {args.dir} (غير متكرر).\nأخرج ONLY مصفوفة JSON [{\"file\":\"\"}]. لا نثر.",
      "output": "json" },
    { "id": "summarize", "type": "map",
      "over": "{steps.discover.json}", "as": "item", "agent": "scout",
      "task": "اقرأ {item.file} واكتب ملخصًا بجملة واحدة.",
      "dependsOn": ["discover"] },
    { "id": "report", "type": "reduce", "from": ["summarize"], "agent": "writer",
      "task": "ادمج في نظرة عامة قصيرة:\n{steps.summarize.output}",
      "dependsOn": ["summarize"], "final": true }
  ]
}
```

1. **`discover`** يسرد كل ملف ويُخرج مصفوفة JSON.
2. **`summarize`** هي `map` — تُوزع وكيلًا فرعيًا واحدًا لكل ملف، محدودة بـ 8 متزامنين، مع `{item.file}` مربوط بكل مسار.
3. **`report`** هي `reduce` — تدمج كل ملخص في نظرة عامة نظيفة واحدة.

الملخصات الوسيطة لا تدخل سياقك أبدًا. بيئة التشغيل تمتلكها؛ تحصل على التقرير. **احفظها مرة واحدة → `/tf:summarize-files dir=src` للأبد.**

### توجيه، بوابة، إعادة محاولة، موافقة، وسقف الإنفاق

```jsonc
{
  "name": "triage-and-fix",
  "budget": { "maxUSD": 1.5 },
  "phases": [
    { "id": "triage", "type": "agent", "agent": "analyst", "output": "json",
      "task": "صنّف البقّ. أخرج ONLY {\"severity\":\"high\"} أو {\"severity\":\"low\"}." },
    { "id": "deep",  "when": "{steps.triage.json.severity} == high", "dependsOn": ["triage"],
      "agent": "executor-code", "task": "حلّل السبب الجذري وأصلحه.",
      "retry": { "max": 2, "backoffMs": 500 } },
    { "id": "quick", "when": "{steps.triage.json.severity} == low",  "dependsOn": ["triage"],
      "agent": "executor-fast", "task": "طبّق الإصلاح السريع." },
    { "id": "approve", "type": "approval", "join": "any", "dependsOn": ["deep", "quick"],
      "task": "راجع الإصلاح قبل إطلاقه." },
    { "id": "ship", "type": "agent", "dependsOn": ["approve"],
      "task": "افتح PR بالتغيير.", "final": true }
  ]
}
```

- **`when`** يُوجّه إلى `deep` *أو* `quick` من JSON التصنيف — الفرع الآخر يُتخطى.
- **`join: "any"`** يسمح لـ `approve` بالإطلاق فور اكتمال أي فرع جرى (وصل OR).
- **`retry`** يعيد تشغيل إصلاح غير مستقر مع تأخير متزايد؛ **`budget`** يوقف التشغيل بأكمله إذا أصبح مكلفًا جدًا.
- **`approval`** يتوقف لإنسان (موافقة / رفض / تحرير) قبل `ship` النهائية.

لا نصوص برمجية. لا `eval`. مجرد بيانات تنفذها بيئة التشغيل — آمنة بما يكفي لتشغيل تعريفات يولدها LLM مباشرة.

## أنواع المراحل

| النوع | ما يفعله | الحقول المطلوبة |
|------|--------------|-----------------|
| `agent` | وكيل فرعي واحد ينفذ مهمة واحدة | `task` |
| `parallel` | تشغيل `branches[]` بشكل متزامن | `branches` (مصفوفة `{task, agent?}`) |
| `map` | **توزيع** على مصفوفة — وكيل فرعي لكل عنصر، `{item}` مربوط | `over`، `task` |
| `gate` | خطوة مراجعة/جودة يمكنها **إيقاف التدفق** | `task` |
| `reduce` | تجميع مخرجات مراحل `from[]` في واحدة | `from`، `task` |
| `approval` | **إدراج الإنسان في الحلقة** — موافقة / رفض / تحرير | — |
| `flow` | تشغيل **تدفق فرعي محفوظ** كمرحلة واحدة (تركيب) | `use` |
| `loop` | **تكرار مهمة حتى الإنجاز** — إعادة تشغيل جسم حتى شرط، تقارب، أو سقف | `task`، `until` |
| `tournament` | **تنافس N من المتغيرات**، حكم يختار الأفضل (أو يجمع) | `task` \| `branches` |

### حقول المرحلة الشائعة

كل مرحلة تحتاج `id` فريد و `type` (الافتراضي `agent`). بالإضافة إلى الحقول الخاصة بكل نوع:

| الحقل | المعنى |
|---|---|
| `agent` | الوكيل للتشغيل (الافتراضي: أول وكيل مكتشف) |
| `dependsOn` | معرفات المراحل التي تنتظرها هذه المرحلة — يبني DAG |
| `join` | `"all"` (الافتراضي) ينتظر كل التبعيات؛ `"any"` هو وصل OR |
| `when` | حارس شرطي — تخطَّ除非 التعبير صحيح |
| `retry` | `{ max, backoffMs?, factor? }` — أعد محاولة وكيل فرعي فاشل |
| `output` | `"text"` (الافتراضي) أو `"json"` (يكشف `{steps.ID.json}`) |
| `model` / `thinking` / `tools` | تجاوزات لكل مرحلة للوكيل الفرعي |
| `cwd` | دليل العمل للوكيل الفرعي |
| `concurrency` | سقف التوزيع لـ `map` / `parallel` (يتجاوز افتراضي التدفق) |
| `final` | يحدد مرحلة حمل النتيجة (وإلا فالمرحلة الأخيرة تفوز) |
| `optional` | الفشل هنا **لا** يوقف التشغيل |
| `use` / `with` | (`flow`) اسم التدفق الفرعي المحفوظ + وسائطه |
| `cache` | `{ scope, ttl?, fingerprint? }` — تخزين مؤقت عبر التشغيلات (انظر أدناه) |

مفاتيح على مستوى التدفق: `name`، `description`، `args`، `concurrency` (الافتراضي 8)، `agentScope`، و `budget: { maxUSD?, maxTokens? }`.

### التحكم في التدفق والموثوقية

- **`when`** — تخطَّ مرحلة ما لم يكن التعبير صحيحًا. يدعم `{refs}`، `== != < > <= >=`، `&& || !`، أقواس، وسلاسل/أرقام مقتبسة. اقرنه بـ `join: "any"` في مرحلة الدمج لتوجيه if/else حقيقي. أخطاء التحليل **تفشل مفتوحة**.
- **`join: "any"`** — وصل OR: المرحلة تعمل بمجرد اكتمال *تبعية واحدة* (الافتراضي `"all"` ينتظر الكل).
- **`retry`** — `{ "max": 2, "backoffMs": 500, "factor": 2 }` يعيد محاولة وكيل فرعي فاشل بتأخير ثابت أو أسي؛ الاستخدام يُجمع وعدد المحاولات يظهر كـ `↻N` في TUI. أخطاء المزود العابرة (حد المعدل / 5xx / مهلة) **تُعاد تلقائيًا حتى بدون سياسة صريحة**؛ الأخطاء الصلبة لا تُعاد.
- **`approval`** — توقف لإنسان (موافقة / رفض / تحرير). الرفض يوقف التدفق؛ التحرير يحقن الملاحظة المكتوبة كمخرج المرحلة للخطوات اللاحقة. التشغيلات غير التفاعلية توافق تلقائيًا.
- **`flow`** — `{ "type": "flow", "use": "deep-research", "with": { "topic": "{item}" } }` يشغّل تدفقًا محفوظًا كمرحلة (الاستدعاء التكراري يُكتشف ويُرفض).

### حلقة حتى الإنجاز (`loop`)

بعض الأعمال متكررة بطبيعتها — صقل مسودة حتى يرضى المراجع، إعادة المحاولة والتحسين حتى تنجح الاختبارات، التقارب على إجابة. مرحلة `loop` تعيد تشغيل جسم مهمة واحدة حتى يتحقق شرط التوقف:

```jsonc
{
  "id": "refine",
  "type": "loop",
  "task": "حسّن هذه المسودة (التكرار {loop.iteration}). المحاولة السابقة:\n{loop.lastOutput}\n\nأخرج JSON {\"draft\":\"…\",\"done\":true|false}.",
  "until": "{steps.refine.json.done} == true",   // مخرجات التكرار نفسها مكشوفة هنا
  "output": "json",
  "maxIterations": 6,        // الافتراضي 10، السقف الصلب 100 — الحلقة تنتهي دائمًا
  "convergence": true        // الافتراضي: توقف مبكرًا إذا كان مخرج التكرار مطابقًا للآخر
}
```

- **المتغيرات المحلية للجسم** — المهمة يمكنها قراءة `{loop.iteration}` (يبدأ من 1)، `{loop.lastOutput}` (مخرج التكرار السابق)، و `{loop.maxIterations}` للبناء على عملها السابق؛ الثلاثة متاحة أيضًا لشرط `until`.
- **`until`** — يُقيّم بعد كل تكرار مع مخرج التكرار مكشوفًا كـ `{steps.<thisId>.output}` / `.json`. نفس عوامل `when`. الحلقة تتوقف بمجرد أن يصبح صحيحًا.
- **تنتهي دائمًا.** أربع نقاط توقف مستقلة: `until` صحيح، **التقارب** (نقطة ثابتة — المخرج مطابق للتكرار السابق)، **`maxIterations`** (سقف صلب عند 100)، أو **تكرار فاشل** (المرحلة تفشل مع الحفاظ على المخرج الجزئي). `until` غير صالح **يوقف** الحلقة بدلاً من الدوران للأبد (آمن في الفشل) ويعرض تحذيرًا على المرحلة.
- TUI يُظهر `↻N` مع سبب التوقف (`done` / `converged` / `max` / `failed`)؛ الاستخدام يُجمع عبر التكرارات. مثل `gate`/`approval`، `loop` **مستبعدة من `cross-run` cache** (كل تشغيل يجب أن يكرر جديدًا).

### البطولة (`tournament`)

للعمل المفتوح، غالبًا ما تأتي أفضل نتيجة من توليد عدة مرشحين واختيار الأقوى — أفضل من N مع حكم، في مرحلة تصريحية واحدة:

```jsonc
{
  "id": "headline",
  "type": "tournament",
  "task": "اكتب عنوانًا جذابًا لمنشور الإطلاق هذا.",
  "variants": 4,                    // أنشئ 4 منافسين لنفس المهمة (الافتراضي 3، الحد الأقصى 20)
  "judge": "اختر العنوان بأقوى خطاف وأوضح وعد.",
  "judgeAgent": "reviewer",          // اختياري؛ الافتراضي هو وكيل المرحلة
  "mode": "best"                     // "best" (الافتراضي) | "aggregate"
}
```

- **المنافسون** — إما `variants: N` نسخ من `task` واحدة (التنوع يأتي من عدم حتمية النموذج)، أو `branches: [{task, agent?}, …]` متميزة عندما تريد أن تتصارع *أساليب مختلفة* ضد بعضها.
- **الحكم** — بعد التوزيع، حكم واحد يرى كل متغير (مرقمًا) بالإضافة إلى معيار `judge` الخاص بك ويختار فائزًا عبر سطر `WINNER: <n>` أو `{"winner": n}`. حكم غير قابل للقراءة **يفشل مفتوحًا** للمتغير 1؛ حكم فاشل يتراجع أيضًا — العمل لا يُفقد أبدًا.
- **`mode`** — `best` يُرجع المتغير الفائز **حرفيًا**؛ `aggregate` يُرجع إجابة الحكم **المركبة** التي تجمع أقوى الأجزاء.
- **دوائر قصيرة:** إذا نجا منافس واحد فقط، يفوز بدون استدعاء حكم؛ إذا فشل الكل، تفشل المرحلة. TUI يُظهر `⚑ N→#k`؛ الاستخدام يجمع المتغيرات + الحكم. مثل `gate`، **مستبعدة من `cross-run` cache**.
- **`budget`** — سقف `{maxUSD, maxTokens}` على مستوى التشغيل؛ بمجرد تجاوزه، المراحل المعلقة تُتخطى والتوزيع الجاري يتوقف عن النشر، منهيًا التشغيل كـ `blocked`.
- **حارس الخمول** — وكيل فرعي يصمت لمدة 5 دقائق يُعتبر عالقًا ويُقتل (SIGTERM → SIGKILL)، لذا لا يمكن لطفل معلق واحد تجميد التدفق بأكمله.

### التخزين المؤقت عبر التشغيلات (`cache`)

كل مرحلة مُعنونة بالمحتوى بالفعل: ضمن **استئناف** تشغيل واحد، المرحلة التي لم تتغير مدخلاتها المحلولة تُتخطى. `cache` يوسع هذا الاستخدام **عبر التشغيلات المستقلة** — إذا قام أي تشغيل سابق بحساب مرحلة بهوية مدخلات مطابقة، تُعاد نتيجتها مقابل **$0.00**.

```jsonc
{
  "id": "analyze-auth",
  "task": "لخّص كيف تعمل وحدة المصادقة.",
  "context": ["src/auth/**/*.ts"],
  "cache": {
    "scope": "cross-run",                 // "run-only" (الافتراضي) | "cross-run" | "off"
    "ttl": "6h",                          // أقصى عمر اختياري قبل اعتبار الضربة خطأ
    "fingerprint": ["git:HEAD", "glob:src/auth/**/*.ts"]  // أدخل حالة العالم في المفتاح
  }
}
```

- **`scope`** — `"run-only"` (الافتراضي) هو السلوك التاريخي بالضبط (استئناف داخل التشغيل فقط). `"cross-run"` يختار المرحلة للتخزين المستمر. `"off"` يعطل إعادة الاستخدام تمامًا (حتى داخل التشغيل)، للتصحيح.
- **النضارة هي اللعبة بأكملها.** مفتاح التخزين المؤقت يتضمن بالفعل المطالبة، وعناصر `over`، وأي ملفات `context` (مقروءة مسبقًا في المهمة). `fingerprint` يُدخل المدخلات *الضمنية* في المفتاح بحيث "العالم تغير" يصبح خطأ في التخزين: `git:HEAD`، `glob:<pat>` (الحجم + وقت التعديل)، `glob!:<pat>` (هاش المحتوى)، `file:<path>`، `env:<NAME>`. `ttl` (`30m`/`6h`/`7d`) هو حاجز زمني.
- **حد صادق:** وكيل فرعي يقرأ ملفًا لم يُعلنه في `context`/`fingerprint` يمكنه مع ذلك تقديم ضربة `cross-run` قديمة. لهذا الافتراضي هو `run-only` ولهذا مراحل `gate`/`approval` **ممنوعة** من `cross-run` (يجب أن تنتج نتيجة جديدة كل تشغيل). اخترها فقط للمراحل التي يكون مخرجها دالة للمدخلات المعلنة.
- التخزين المؤقت موجود في `.pi/taskflows/cache/` (متجاهل بـ git). امسحه بـ `action: "cache-clear"`. الأساس المنطقي الكامل: [`docs/rfc-cross-run-memoization.md`](./docs/rfc-cross-run-memoization.md).

### مراحل البوابة (مراقبة الجودة)

`gate` تشغّل وكيلًا لمراجعة المخرجات العلوية ويمكنها **منع بقية سير العمل.** أنهِ مهمة البوابة بطلب حكم يمكن لبيئة التشغيل قراءته:

- سطر نهائي `VERDICT: PASS` أو `VERDICT: BLOCK` (يقبل أيضًا `OK`، `FAIL`، `STOP`، `REJECT`، `HALT` — آخر ظهور يفوز)، أو
- JSON مثل `{"continue": false, "reason": "فحوصات المصادقة مفقودة"}` / `{"verdict": "block", "reason": "..."}`.

عند **BLOCK**، المراحل اللاحقة تُتخطى والتشغيل ينتهي كـ `blocked` مع عرض السبب. **المخرجات الغامضة تفشل مفتوحة** (تُعامل كـ PASS) — البوابة لا توقف تدفقك أبدًا عن طريق الصدفة.

```
راجع التدقيق أدناه. إذا كانت أي نقطة نهاية تفتقد المصادقة، أنهِ بـ
"VERDICT: BLOCK" وسبب في سطر واحد؛ وإلا أنهِ بـ "VERDICT: PASS".

{steps.audit.output}
```

## الاستيفاء والتعبيرات

| العنصر النائب | يحل إلى |
|---|---|
| `{args.X}` | وسيطة الاستدعاء |
| `{steps.ID.output}` | مخرج نص مرحلة سابقة |
| `{steps.ID.json}` | مخرج سابق محلل كـ JSON (أو `{steps.ID.json.field}`) |
| `{item}` / `{item.field}` | العنصر الحالي داخل مرحلة `map` |
| `{previous.output}` | مخرج المرحلة التي تسبق مباشرة |

قواعد الشرط (لـ `when`): `== != < > <= >=`، `&& || !`، أقواس، سلاسل/أرقام مقتبسة، وأي مرجع `{...}` — مثلاً `"when": "{steps.triage.json.route} == deep && {args.force} != true"`.

> الإشارة إلى `{steps.X}` غير المعلنة في `dependsOn` هي **خطأ تحقق صارم** — بيئة التشغيل تلتقط أكثر أخطاء الـ pipeline شيوعًا قبل تشغيل وكيل واحد.

## الأوامر

التدفقات المحفوظة تصبح اختصارات CLI. جميع الأوامر تعمل في جلسة Pi:

| الأمر | ما يفعله |
|---|---|
| `/tf list` | سرد جميع التدفقات المحفوظة |
| `/tf run <name> [args]` | تشغيل تدفق محفوظ (مثال: `/tf run summarize-files dir=src`) |
| `/tf show <name>` | طباعة تعريف التدفق |
| `/tf runs` | تصفح سجل التشغيلات الأخيرة (واجهة TUI تفاعلية) |

| `/tf init` | **ربط أدوار النماذج تفاعليًا** بنماذجك الممكّنة (يكتب `~/.pi/agent/settings.json`) |

إجراءات الأداة (التي يستخدمها النموذج): `run` (تعريف `define` مضمن أو `name` محفوظ)، `save`، `resume`، `list`، `init`.

## الاستئناف عبر الجلسات

## الوكلاء

يحتوي Taskflow على **18 وكيلًا مدمجًا** — كل واحد هو ملف `.md` مع مطالبة نظام مضبوطة، ومستوى تفكير، ومجموعة أدوات. يمكنك الإشارة إليهم بالاسم `name` في أي مرحلة أو اختصار، فور التثبيت. لا حاجة لإعداد.

### قائمة الوكلاء المدمجين

| الوكيل | الدور | التفكير | الدور الافتراضي |
|---|---|---:|---|

| `executor-fast` | إصلاحات تافهة (≤2 ملف، ≤50 سطر) | off | `{{fast}}` |
| `executor-code` | تنفيذ معقد متعدد الملفات | high | `{{strong}}` |
| `executor-ui` | تغييرات الواجهة الأمامية / التصميم / الظاهرية | high | `{{vision}}` |
| `scout` | استكشاف قاعدة الشيفرة بسرعة | off | `{{fast}}` |
| `planner` | إنشاء خطة التنفيذ | high | `{{strong}}` |
| `analyst` | تحليل المتطلبات، اكتشاف الغموض | high | `{{thinker}}` |
| `critic` | شك ذاتي مضمن أثناء الاستدلال | xhigh | `{{thinker}}` |
| `reviewer` | مراجعة عامة للشيفرة / البنية | high | `{{strong}}` |
| `risk-reviewer` | مخاطر الخلفية / البنية التحتية / قاعدة البيانات / API | high | `{{reasoner}}` |
| `security-reviewer` | ثغرات أمنية، مصادقة/تشفير | xhigh | `{{reasoner}}` |
| `plan-arbiter` | بوابة جودة الخطة (المهام المعقدة) | high | `{{arbiter}}` |
| `final-arbiter` | كسر التعادل عندما يختلف النقاد | xhigh | `{{arbiter}}` |
| `test-engineer` | تصميم وتنفيذ الاختبارات | high | `{{fast}}` |
| `doc-writer` | كتابة الوثائق | off | `{{fast}}` |
| `recover` | استرداد الجلسة بعد الضغط | low | `{{fast}}` |
| `verifier` | تشغيل الاختبارات، التحقق من النتائج | off | `{{fast}}` |
| `visual-explorer` | تحليل بيانات Figma الوصفية | high | `{{vision}}` |

الوكلاء متعددو الطبقات: **مدمج → مستخدم (`~/.pi/agent/agents/`) → مشروع (`.pi/agents/`)**. وكيل مستخدم أو مشروع بنفس الاسم `name` يتجاوز المدمج — لذا يمكنك تخصيص أي وكيل دون لمس الحزمة.

### أدوار النموذج

حقل `model` لكل وكيل مدمج يستخدم **عنصرًا نائبًا للدور** (مثل `{{fast}}`) بدلاً من سلسلة مزود ثابتة. هذا يفصل *القصد* عن *التنفيذ* — تقوم بتعيين الأدوار للنماذج مرة واحدة، وكل وكيل يتكيف.

| الدور | القصد | النموذج النموذجي |
|---|---|---|
| `{{fast}}` | رخيص وسريع — حجم كبير، مخاطر منخفضة | DeepSeek V4 Flash |
| `{{strong}}` | متوازن — تخطيط، مراجعة، تعقيد متوسط | MiMo v2.5 Pro |
| `{{thinker}}` | تحليل عميق — متطلبات، نقد | DeepSeek V4 Pro |
| `{{arbiter}}` | حكم نهائي — كسر التعادل، بوابات جودة الخطة | Qwen 3.7 Max |
| `{{vision}}` | متعدد الوسائط — عمل واجهة المستخدم، قراءة التصميم | MiniMax M3 |
| `{{reasoner}}` | استدلال حذر — أمن، مخاطر | GLM 5.1 |

بدون تكوين، الوكلاء يتراجعون إلى النموذج الافتراضي لـ Pi. لتعيين الأدوار لنماذج حقيقية، شغّل الإعداد التفاعلي:

```bash
/tf init
```

`/tf init` يبدأ بـ **قائمة إجراءات**. المستخدمون الجدد يحصلون على اختصار من خيارين ("استخدم الإعدادات الافتراضية الموصى بها" / "تكوين كل دور"). المستخدمون العائدون يرون القائمة الكاملة من 5 خيارات:

```
? ماذا تريد أن تفعل بأدوار النموذج؟
  ❯ استخدم الإعدادات الافتراضية الموصى بها
    تكوين كل دور
    تعديل دور واحد
    عرض الأدوار الحالية
    إلغاء
```

المنتقي يُظهر **الأسماء المعروضة** للنماذج مع علامات القدرة ومؤشرات الحالي/الموصى به:

```
? النموذج لـ 'vision' — متعدد الوسائط (executor-ui، visual-explorer)
  الحالي: openrouter/anthropic/claude-sonnet-4-6
  الموصى به: minimax/MiniMax-M3
  ───────────────
  ❯ MiniMax M3 (minimax/MiniMax-M3) · صورة ✓ · استدلال ✓ · (موصى به)
    Claude Sonnet 4.6 (openrouter/anthropic/...) · صورة ✓ · استدلال ✓ · (حالي)
    GPT-5 (openrouter/openai/gpt-5) · صورة ✓
    DeepSeek V4 Flash (openrouter/deepseek/v4-flash)
    ───────────────
    مخصص (اكتب بنفسك)
    احتفظ بالحالي
    العودة إلى قائمة الإجراءات
```

قبل الحفظ، **شاشة معاينة** تُظهر الفرق بين تغييراتك:

```
? مراجعة التغييرات:
  fast       openrouter/deepseek/deepseek-v4-flash   (لم يتغير)
  strong     openrouter/xiaomi/mimo-v2.5-pro         (لم يتغير)
  thinker    openrouter/qwen/qwen3.7-max             (تغير ← كان: openrouter/deepseek/v4-pro)
  arbiter    openrouter/qwen/qwen3.7-max             (لم يتغير)
  vision     minimax/MiniMax-M3                      (لم يتغير)
  reasoner   z-ai/glm-5.1                            (لم يتغير)
  ───────────────
  ❯ احفظ هذه التغييرات
    تعديل دور
    إلغاء
```

خياراتك تُكتب إلى `~/.pi/agent/settings.json`:

```json
{
  "modelRoles": {
    "fast":     "openrouter/deepseek/deepseek-v4-flash",
    "strong":   "openrouter/xiaomi/mimo-v2.5-pro",
    "thinker":  "openrouter/deepseek/deepseek-v4-pro",
    "arbiter":  "openrouter/qwen/qwen3.7-max",
    "vision":   "minimax/MiniMax-M3",
    "reasoner": "z-ai/glm-5.1"
  }
}
```

حرّر القيم يدويًا في أي وقت، أو أعد تشغيل `/tf init`. يمكنك أيضًا تجاوز وكلاء فرديين عبر `subagents.agentOverrides` في نفس الملف:

```json
{
  "modelRoles": { ... },
  "subagents": {
    "agentOverrides": {
      "executor": { "model": "anthropic/claude-sonnet-4-20250514" },
      "reviewer": { "thinking": "xhigh" }
    }
  }
}
```

### مسار الأداة (`action="init"`)

يمكن للنموذج أيضًا تكوين الأدوار عبر أداة `taskflow`:

| الوضع | السلوك |
|---|---|
| `mode: "show"` (الافتراضي) | تقرير للقراءة فقط للأدوار الحالية `modelRoles`. لا يكتب أبدًا. |
| `mode: "apply-defaults"` + `force: true` | يكتب `RECOMMENDED_DEFAULTS` إلى `settings.json`، مع الحفاظ على المفاتيح القديمة. |
| `mode: "interactive"` | يُطلق قائمة الإجراءات الكاملة + تدفق المنتقي (يتطلب جلسة UI). |

> **ملاحظة إهمال v0.0.13:** إذا تم حذف `mode`، تتراجع الأداة إلى سلوك v0.0.12 عندما يكون `modelRoles` فارغًا (يكتب الإعدادات الافتراضية تلقائيًا) مع إشعار إهمال عبر `console.warn`. إذا كان `modelRoles` موجودًا بالفعل، تتصرف كـ `mode: "show"`. سيتم إزالة هذا الجسر في v0.0.14.

### الوكلاء المخصصون

ضع ملف `.md` في `~/.pi/agent/agents/` (على مستوى المستخدم) أو `.pi/agents/` (على مستوى المشروع، شاركه بالـ commit) لإضافة وكيلك الخاص:

```markdown
---
name: my-linter

description: شغّل ESLint وأبلغ عن المخالفات

tools: read, bash

model: "{{fast}}"

thinking: off
---

أنت وكيل فحص. شغّل `npx eslint --format json` على
الملفات المقدمة. أبلغ عن المخالفات مجمعة حسب الملف. لا إصلاحات.
```

ثم أشر إليه في أي مرحلة: `{ "agent": "my-linter", "task": "افحص src/" }`.

## ما في الداخل

<div align="center">

**0 تبعيات تشغيل** · **394 اختبارًا** · **10 أنواع مراحل** · **استئناف عبر الجلسات** · **تخزين مؤقت عبر التشغيلات** · **~4.9k سطر بيئة تشغيل**

</div>

- **صفر تبعيات تشغيل.** لا حقل `dependencies` — بيئة التشغيل مبنية بالكامل على مدمجات Node (`fs` / `path` / `os` / `child_process` / `crypto`). قفل الملف هو `fs.openSync("wx")`، ليس مكتبة طرف ثالث.
- **371 اختبارًا عبر 14 مجموعة** تغطي التزامن، قفل الملف الذري (انحدارات سباق 8 عمليات)، تحصين مسار العبور، استئناف عبر الجلسات، نضارة التخزين المؤقت عبر التشغيلات (عزل مفتاح التدفق/التفكير/الأدوات، إبطال البصمة، إزالة TTL/LRU)، أحكام البوابة، سقف الميزانية، إعادة المحاولة/التأخير، تدفقات الموافقة، إنهاء الحلقة، تحكيم البطولة، تركيب التدفق الفرعي، عزل الاستدعاء، حارس الخمول، تكوين init لأدوار النموذج، و parseModelFromLabel مع انحدار اسم النموذج بين قوسين — بالإضافة إلى اختبار شامل مباشر يشغّل وكلاء فرعيين حقيقيين و dogfood للتخزين المؤقت عبر التشغيلات.
- **مقوّى بالتصميم.** دفاع ضد مسار العبور (معجمي + `realpath`)، التحقق من صحة runId، تعقيم HTML/الأخطاء، كتابات ذرية، سرقة القفل العالق عبر `rename`، وحارس خمول يقتل الوكلاء الفرعيين العالقين.
- **يُستخدم كعلف خاص (dogfooded).** كل ميزة جديدة يجب أن تنجو من تدفق `self-improve` الخاص بالمشروع قبل أن تُطلق.

## 🍽️ نأكل من طعام كلابنا الخاص

كل ميزة في `pi-taskflow` تُطلق **عبر `pi-taskflow`.**

تدفق `self-improve` الخاص بنا هو DAG من 10 مراحل — يدقق قاعدة الشيفرة، ويصلح العيوب، ويتحقق من الصحة، ويُقيم البوابة على الجودة، ويُظهر التقرير — كل هذا بشكل تصريحي. إنه محفوظ كـ `/tf:self-improve` ويُشغّل قبل كل إصدار. لا يوجد منظّم وكلاء آخر في نظام Pi البيئي يبني نفسه بنفسه.

| الحملة | النطاق | المراحل | النتيجة |
|----------|-------|--------|---------|
| [dogfood v0.0.8](./docs/dogfooding-v0.0.8-report.md) | تدقيق شامل لقاعدة الشيفرة → فرز → إصلاح → تحقق | 10 مراحل، 234 اختبارًا | 13 إصلاحًا، كلها ناجحة |
| [التدقيق الذاتي v0.0.6](./docs/self-audit-report.md) | جرد → خريطة تدقيق → بوابة → موافقة → خريطة إصلاح → تجميع | 9 مراحل | 11 عيبًا خطيرًا تم إصلاحه |
| [dogfood التخزين المؤقت عبر التشغيلات](./docs/rfc-cross-run-memoization.md) | بيئة تشغيل حقيقية + تخزين على القرص | منصة اختبار مخصصة | صحة التخزين المؤقت تحت بصمات خصومة |
| [مراجعة متقاطعة خصومة](./docs/brainstorm-adversarial-review-report.md) | مراجعة خصومة متعددة الوكلاء | `tournament` + `gate` | تم إطلاق إصلاح P0 لمفتاح التخزين المؤقت |
| [مراجعة إعادة تصميم Init](./docs/issue-necessity-review-report.md) | تدقيق الضرورة → فحوصات متوازية → حكم | 7 مراحل | تم التحقق من خطة إعادة تصميم كاملة |

> **تلويح ميتا:** استخدمنا `pi-taskflow` لتوزيع `map`، وأحكام `gate`، وموافقة `approval` لإدراج الإنسان، وبطولة `tournament` لأفضل من N، وحلقة `loop` حتى الإنجاز، وتخزين `cross-run` مؤقت — لبناء `pi-taskflow`.

## الحالة والحدود

**v0.0.13** — حلقة حتى الإنجاز (مرحلة `loop`: كرّر حتى شرط، تقارب، أو سقف)، بطولة (أفضل من N مع حكم)، تخزين مؤقت عبر التشغيلات (ذاكرة مؤقتة معنونة بالمحتوى مع بصمات git/ملف/glob/متغير بيئة و TTL)، `/tf init` تفاعلي مع منتقي نماذج مدرك للأدوار + معاينة الفرق + كتابة دمج ذرية، 18 وكيلًا مدمجًا مع 6 أدوار نموذج. طبقة التحكم في التدفق والموثوقية الكاملة (حراس `when`، وصل `join: any`، إعادة محاولة `retry`/تأخير، موافقة `approval`، تركيب `flow`، سقف `budget`، حارس الخمول) فوق DSL + بيئة تشغيل DAG (`agent`/`parallel`/`map`/`gate`/`reduce`). تدفقات مضمنة ومحفوظة، استئناف عبر الجلسات، تقدم مباشر، وسياق معزول. التشغيل ينفذ كاستدعاء أداة واحد متدفق.

حدود معروفة (مُتتبّعة، محدودة — لا مفاجآت أثناء التدفق):

- **لا تنفيذ خلفي منفصل.** التشغيل يحتاج جلسة Pi مفتوحة. التنفيذ الخلفي الحقيقي (ومشغّلات الأحداث/المهام المجدولة فوقه) على خارطة الطريق.
- **لا `output: "file"`.** المخرجات نص/JSON فقط — اكتب الملفات عبر استدعاء أداة `write` للوكيل.
- **`map` يتطلب مصفوفة JSON.** حقل `over` يجب أن يحل إلى مصفوفة `{steps.ID.json}`. لفّ قائمة نصية في مرحلة `output: "json"` لوكيل واحد أولاً.
- **DAG يجب أن يكون غير دوري.** الحلقات تُرفض عند التحقق.
